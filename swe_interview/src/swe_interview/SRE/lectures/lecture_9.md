
## О чем лекция
Лекция посвящена стратегиям разделения больших объемов данных на шарды, различным подходам к шардированию (на примере Google), управлению виртуальными партициями и равномерному распределению нагрузки по серверам.

## Ключевые вопросы
- Когда данных становится слишком много для одной БД?
- Как распределять данные по шардам и серверам?
- Какие существуют стратегии шардирования для разных типов запросов?
- Как обеспечить равномерное распределение и избежать горячих точек?
- Чем шардирование отличается от репликации?

## Основная часть

### Основы шардирования

**Проблема**: Данных слишком много, они не помещаются в одну БД.

**Решение**: Разделить (partition) данные и хранить в разных местах.

```
Исходная БД:
[1000 TB данных] → Не помещается!

После шардирования:
[Сервер 1: 100 TB]
[Сервер 2: 100 TB]
[Сервер 3: 100 TB]
...
[Сервер 10: 100 TB]
```

**Пример распределения по строкам**:
```sql
-- Шард 1
Users WHERE id % 10 = 0

-- Шард 2
Users WHERE id % 10 = 1

-- ... и т.д.
```

---

### Проблемы шардирования

| Проблема | Описание |
|----------|----------|
| **Межшардовые запросы** | Запросы, затрагивающие несколько шардов, медленнее из-за сетевых взаимодействий |
| **Поиск данных** | Сложнее найти нужные данные — нужно знать, на каком шарде они лежат |
| **JOIN-ы** | JOIN между таблицами на разных шардах очень дорогие |
| **Транзакции** | Распределенные транзакции между шардами сложны в реализации |

**Когда делают шардирование**:
- ✅ Для **очень больших данных** (сотни TB, PB)
- ✅ Для **увеличения надежности** (потеря части данных не критична)

---

### Шардирование + Репликация

**Best practice**: Комбинировать шардирование с репликацией.

```
Шард 1 → [Master] → [Replica 1] → [Replica 2]
Шард 2 → [Master] → [Replica 1] → [Replica 2]
Шард 3 → [Master] → [Replica 1] → [Replica 2]
```

**Преимущества**:
- Повышение доступности на чтение (читаем из реплик)
- Отказоустойчивость (если упал один шард-мастер, есть реплики)
- Распределение нагрузки

### Виртуальные партиции

**Проблема**: Сложно перераспределять данные при добавлении новых серверов.

**Решение**: Создать **много виртуальных партиций**, затем распределить их по физическим серверам.

```
Виртуальные партиции: 1000 шардов
Физические серверы: 10 серверов

Сервер 1 → [Partition 1, 11, 21, ..., 991]     (100 партиций)
Сервер 2 → [Partition 2, 12, 22, ..., 992]     (100 партиций)
...
Сервер 10 → [Partition 10, 20, 30, ..., 1000]  (100 партиций)
```

**Преимущества**:
- При добавлении нового сервера — просто перемещаем виртуальные партиции
- Не нужно перехешировать все данные
- Более гибкое управление нагрузкой

**С репликацией**:
```
Partition 1:
  - Копия 1 на Сервере A
  - Копия 2 на Сервере B

Partition 2:
  - Копия 1 на Сервере B
  - Копия 2 на Сервере C
```

### Ключи шардирования (Shard Keys)

**Задача**: Определить, как распределять документы по шардам.

**Обычно используют Primary Key**:
```python
shard_id = hash(primary_key) % num_shards
```

**Критически важно**:
- ⚠️ Ключ должен быть **уникальным** (иначе конфликты)
- ⚠️ Избегать дубликатов
- ⚠️ Распределение должно быть **равномерным**

**Плохие ключи шардирования**:
- `country_code` — неравномерное распределение (в США больше пользователей)
- `created_date` — новые данные всегда попадают на один шард

**Хорошие ключи шардирования**:
- `user_id` (UUID)
- `hash(email)`
- Составной ключ: `hash(user_id + timestamp)`

### Стратегии поиска в шардированных системах

#### Проблема масштаба

**Сценарий**: У нас **12 000 серверов** с репликациями. Как искать данные?

#### Решение: Координирующий сервер (Scatter-Gather)

```
[Клиент]
   ↓
[Координирующий сервер]
   ↓
   ├→ [Шард 1] → Топ 10 результатов
   ├→ [Шард 2] → Топ 10 результатов
   ├→ [Шард 3] → Топ 10 результатов
   ...
   └→ [Шард 12000] → Топ 10 результатов
   ↓
[Мердж 12000 списков → Итоговый Топ 10]
   ↓
[Клиент]
```

**Алгоритм**:
1. Координатор отправляет запрос **всем шардам** параллельно
2. Каждый шард ищет у себя и возвращает **Top-N** результатов
3. Координатор **мерджит** все результаты и выбирает финальный Top-N

**Пример** (поисковый запрос):
```
Запрос: "machine learning"

Шард 1 → [doc123: score=0.95, doc456: score=0.89, ...]
Шард 2 → [doc789: score=0.92, doc234: score=0.87, ...]
...

Координатор → Мердж по score → [doc123, doc789, doc456, ...]
```

**Так работало в Google Search!**

### Стратегии шардирования из Google

#### Стратегия 1: DOC-SHARDED

**Принцип**: Шардирование по **документам** (целиком).

```
Шард 1: [doc1, doc4, doc7, doc10, ...]
Шард 2: [doc2, doc5, doc8, doc11, ...]
Шард 3: [doc3, doc6, doc9, doc12, ...]
```

**Как распределять**:
```python
shard_id = hash(document_id) % num_shards
```

**Поиск**:
```
Запрос: "как кормить котика"

1. Отправить запрос на ВСЕ шарды
2. Каждый шард ищет в своих документах
3. Каждый шард возвращает Top-10
4. Координатор мерджит → финальный Top-10
```

**Плюсы**:
- ✅ Простая логика
- ✅ Равномерное распределение нагрузки

**Минусы**:
- ⚠️ Нужно обращаться ко **всем шардам** на каждый запрос

---

#### Стратегия 2: TERM-SHARDED (Inverted Index Sharding)

**Принцип**: Шардирование по **терминам** (словам).

```
Запрос: "как кормить котика"

Шард "как"      → [doc1, doc5, doc9, doc100, ...]
Шард "кормить"  → [doc5, doc23, doc45, ...]
Шард "котика"   → [doc5, doc89, doc234, ...]
```

**Поиск**:
```
1. Определить, на каких шардах лежат слова "как", "кормить", "котика"
2. Обратиться только к этим шардам (не ко всем!)
3. Получить списки документов для каждого слова
4. Найти пересечение (документы со всеми словами)
5. Ранжировать и вернуть Top-N
```

**Плюсы**:
- ✅ Обращаемся только к **нужным шардам** (не ко всем 12000!)
- ✅ Меньше сетевого трафика

**Минусы**:
- ⚠️ Сложная логика мерджа (нужно искать пересечения)
- ⚠️ Неравномерная нагрузка (частые слова типа "the", "a" создают горячие шарды)

**Пример распределения**:
```
hash("как")     % 1000 = 42  → Шард 42
hash("кормить") % 1000 = 157 → Шард 157
hash("котика")  % 1000 = 893 → Шард 893
```

**Так тоже работало в Google!**

### Практический пример: Аналитика событий

**Задача**: Хранить миллионы событий в день, делать аналитику.

**Решение с MySQL**: Создавать **таблицу на каждый день**.

```sql
CREATE TABLE events_2024_01_15 (
  id BIGINT,
  user_id INT,
  event_type VARCHAR(50),
  timestamp TIMESTAMP,
  ...
);

CREATE TABLE events_2024_01_16 (...);
CREATE TABLE events_2024_01_17 (...);
-- И так далее
```

**Запрос за один день**:
```sql
SELECT COUNT(*) FROM events_2024_01_15
WHERE event_type = 'click';
```

**Запрос за диапазон дат**:
```sql
SELECT SUM(cnt) FROM (
  SELECT COUNT(*) as cnt FROM events_2024_01_15 WHERE event_type = 'click'
  UNION ALL
  SELECT COUNT(*) as cnt FROM events_2024_01_16 WHERE event_type = 'click'
  UNION ALL
  SELECT COUNT(*) as cnt FROM events_2024_01_17 WHERE event_type = 'click'
) t;
```

**Ограничение**:
- ⚠️ Невозможно запросить за **больше года** (слишком тяжелый запрос)
- ⚠️ Нужно явно перечислять все таблицы

**Современное решение**: Партиционирование в PostgreSQL или использование колоночных БД (ClickHouse).

### Колоночные базы данных

**Принцип**: Данные хранятся по **столбцам**, а не по строкам.

**Традиционная (построчная) БД**:
```
Row 1: [id=1, name=Alice, age=25, city=NYC]
Row 2: [id=2, name=Bob,   age=30, city=LA]
Row 3: [id=3, name=Carol, age=35, city=SF]
```

**Колоночная БД**:
```
Column id:   [1, 2, 3]
Column name: [Alice, Bob, Carol]
Column age:  [25, 30, 35]
Column city: [NYC, LA, SF]
```

**Шардирование по столбцам**:
```
Сервер 1: [id, timestamp]
Сервер 2: [user_id, event_type]
Сервер 3: [payload, metadata]
```

**При запросе**:
```sql
SELECT user_id, event_type
FROM events
WHERE timestamp > '2024-01-01';
```

Обращаемся только к:
- Серверу 1 (для фильтрации по `timestamp`)
- Серверу 2 (для получения `user_id`, `event_type`)

**Преимущества**:
- ✅ Высокая степень сжатия (столбцы однородны)
- ✅ Быстрые агрегации
- ✅ Читаем только нужные столбцы

**Примеры**: ClickHouse, Apache Druid, Vertica

### Принципы распределения шардов

**Задача**: У нас **4 документа** и **15 виртуальных шардов**. Как распределить по серверам?

#### Подход 1: Динамическое разделение (Split)

**Алгоритм**:
1. Задаем **максимальное количество шардов на сервер** (например, 5)
2. Когда сервер заполнился → **разделяем** его пополам
3. Половину шардов переносим на новый сервер

```
Начальное состояние:
Сервер 1: [Shard 1, 2, 3, 4, 5] ← Заполнен!

После split:
Сервер 1: [Shard 1, 2, 3]
Сервер 2: [Shard 4, 5]

Добавляем новые данные:
Сервер 1: [Shard 1, 2, 3, 6, 7] ← Заполнен!
Сервер 2: [Shard 4, 5]

После split:
Сервер 1: [Shard 1, 2, 3]
Сервер 2: [Shard 4, 5]
Сервер 3: [Shard 6, 7]
```

#### Подход 2: Равномерное распределение

**Алгоритм**:
1. Задаем **максимальное количество шардов** (N)
2. **Равномерно** распределяем по доступным серверам

```
15 шардов, 3 сервера:

Сервер 1: [1, 4, 7, 10, 13]    (5 шардов)
Сервер 2: [2, 5, 8, 11, 14]    (5 шардов)
Сервер 3: [3, 6, 9, 12, 15]    (5 шардов)
```

**Формула**:
```python
num_shards_per_server = total_shards // num_servers
server_id = shard_id % num_servers
```

**Цель**: Избежать ситуации, когда **много серверов простаивают**, а несколько перегружены.

### Кастомное шардирование vs Встроенное

| Аспект | Шардирование | Репликация |
|--------|--------------|------------|
| **Реализация** | ⚠️ Обычно **пишут сами** | ✅ Обычно **встроено в БД** |
| **Сложность** | Сложно, зависит от бизнес-логики | Проще, стандартные решения |
| **Гибкость** | Полный контроль | Ограничена возможностями БД |

**Почему шардирование пишут сами**:
- Каждый бизнес-случай уникален
- Характер запросов сильно влияет на выбор стратегии
- Встроенные решения БД часто недостаточно гибкие

**Примеры кастомного шардирования**:
- Google Search (DOC-SHARDED + TERM-SHARDED)
- Instagram (по `user_id`)
- Discord (по `guild_id`)

## Выводы и инсайты

- **Шардирование ≠ Репликация**:
  - Репликация — для отказоустойчивости и масштабирования чтения
  - Шардирование — для масштабирования объема данных и записи

- **Виртуальные партиции — ключ к гибкости**: Позволяют легко добавлять/удалять серверы без перехеширования всех данных

- **Scatter-Gather работает**: Даже при 12 000 серверов можно эффективно искать, отправляя запросы параллельно

- **Выбор стратегии зависит от запросов**:
  - DOC-SHARDED — когда запросы затрагивают много документов
  - TERM-SHARDED — когда запросы специфичны (поиск по ключевым словам)

- **Колоночные БД идеальны для аналитики**: Читаем только нужные столбцы, отличное сжатие

- **Равномерность критична**: Неравномерное распределение шардов → горячие точки и простаивающие серверы

- **Шардирование требует кастомизации**: Универсального решения нет, нужно адаптировать под конкретную задачу

- **Начните с репликации**: Шардирование добавляет огромную сложность. Используйте только когда действительно необходимо (PB данных, миллионы RPS)

- **Планируйте ключи шардирования заранее**: Изменить схему шардирования на работающей системе — кошмар

- **Мониторинг балансировки**: Регулярно проверяйте, что шарды равномерно распределены и не перегружены

- **Google доказал**: Обе стратегии (DOC-SHARDED и TERM-SHARDED) работают на огромных масштабах, выбор зависит от паттернов использования
