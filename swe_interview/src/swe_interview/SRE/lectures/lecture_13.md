## О чем лекция
Лекция посвящена практическим аспектам обнаружения сбоев, правильной организации мониторинга, философии алертирования и механизмам доставки критических уведомлений инженерам. Лектор разбирает реальный недавний инцидент и показывает, как правильный мониторинг мог бы сэкономить часы времени.

## Ключевые вопросы
- Как правильно регистрировать метрики, чтобы не пропустить сбой?
- Почему недостаточно просто считать количество запросов?
- Какие алерты действительно нужны, а какие создают шум?
- Как организовать доставку алертов, чтобы инженеры гарантированно реагировали?
- Кто должен дежурить и как распределять нагрузку?

## Основная часть
### Кейс: Реальный недавний сбой
#### Ситуация
**Проблема**: Микросервис не работает, пользователи жалуются, но мониторинг показывает это:

```
Количество запросов
    │
    │ ▬▬▬▬▬▬▬▬▬▬▬▬
    │
    │
    │ _______________  ← Упало до нуля
    └─────────────────→ Время
```

**Первая гипотеза команды**: "К нам не идут запросы → проблема ДО нашего сервиса"

**Действия**:
1. Вызванивали всех, кто перед микросервисом:
   - DDoS защита
   - Load balancer
   - Сетевые сегменты
2. Все проверяют: "К вам запросы идут точно!"
3. **Потрачено: 30 минут**

#### В чем была ошибка мониторинга?
**Вопрос**: В какой момент считались эти запросы?
**Проблема**: Запросы считались **после начала обработки**, а не при получении.

```python
# ❌ НЕПРАВИЛЬНО - считаем поздно
@app.route('/api/users')
def get_users():
    # ... какая-то логика ...
    
    # Где-то в середине/конце
    metrics.record("requests_total", 1)  # Поздно!
    
    return users
```

**Что происходило на самом деле**:
1. Запросы приходили в приложение
2. Приложение пыталось начать работу
3. **Не могло залогиниться / инициализироваться**
4. До основного кода не доходило
5. Метрика запросов не регистрировалась

**Правильный подход**:
```python
# ✅ ПРАВИЛЬНО - считаем СРАЗУ
@app.route('/api/users')
def get_users():
    # Строка #1: регистрируем факт получения запроса
    metrics.record("requests_received", 1)
    log_request_start(request_id=gen_id())
    
    # Теперь обрабатываем
    try:
        result = process_request()
        metrics.record("requests_success", 1)
        return result
    except Exception as e:
        metrics.record("requests_error", 1)
        raise
```

#### Почему график не был пустым полностью?
**Особенность микросервиса**: Много разных эндпоинтов (ручек).
**Что показывал график**:
```
Общие запросы:     ▬▬▬▬▬______ (упал почти до нуля)
Попытки входа:     ▬▬▬▬▬▬▬▬▬▬ (продолжались!)
```

**Проблема**: 
- Попытки входа приходили
- Вход не удавался
- До остальных эндпоинтов не доходило
- График показывал только частичный трафик

**Решение**: Отдельный график для критичных операций (login, initialization).

### Вторая часть сбоя: Проблема с базой данных
**После 30 минут** выяснили, что запросы идут, но сервис не работает.
**Проблема**: База данных не отвечала.

```
[Микросервис] → [База данных] ← ???
                      ↓
                Нет ответа
```

**Симптомы**:
- Либо timeout
- Либо пустые ответы
- Либо какие-то ошибки

**Что не было в мониторинге**: RED метрики для исходящих запросов в БД!
**Потрачено еще**: 40 минут на поиск проблемы
**Решение**: Перезапустили БД → все заработало
**Итого потрачено**: 30 + 40 = **70 минут** 😱

### RED метрики: Правильная регистрация
#### Входящие запросы (Inbound RED)
**Что регистрируем**:

```python
# 1. Requests - при ПОЛУЧЕНИИ запроса
@app.before_request
def log_request_start():
    request_id = generate_request_id()
    g.request_id = request_id
    g.start_time = time.time()
    
    # Строка #1 - запрос получен!
    metrics.record("http_requests_received", 1, labels={
        "method": request.method,
        "endpoint": request.endpoint
    })
    
    # В память для отслеживания таймаутов
    active_requests[request_id] = {
        "start_time": g.start_time,
        "endpoint": request.endpoint
    }

# 2. Errors / Success - при ЗАВЕРШЕНИИ
@app.after_request
def log_request_end(response):
    duration = time.time() - g.start_time
    
    # Удаляем из активных
    active_requests.pop(g.request_id, None)
    
    if response.status_code < 400:
        metrics.record("http_requests_success", 1)
    else:
        metrics.record("http_requests_error", 1, labels={
            "status": response.status_code
        })
    
    # 3. Duration
    metrics.record("http_request_duration", duration)
    
    return response

# Обнаружение таймаутов
def check_timeouts():
    """Фоновый процесс для обнаружения зависших запросов"""
    while True:
        now = time.time()
        
        for request_id, info in active_requests.items():
            elapsed = now - info["start_time"]
            
            if elapsed > REQUEST_TIMEOUT:
                # Запрос завис!
                metrics.record("http_requests_timeout", 1, labels={
                    "endpoint": info["endpoint"]
                })
                
                # Опционально: можно попытаться ответить клиенту
                # (если код позволяет)
        
        time.sleep(10)  # Проверяем каждые 10 секунд
```

**Два графика для анализа**:

```
График 1: Requests Received (начались)
    │ ▬▬▬▬▬
    │      ╲____
    └─────────────→

График 2: Requests Completed (завершились)
    │   ▬▬▬▬▬
    │        ╲____  ← Смещен на duration
    └─────────────→
```

**Нормальная ситуация**: График 2 отстает от графика 1 на **среднюю длительность запроса*

**Сбой**: 
- График 1 упал → запросы перестали приходить
- График 2 упал позже → обрабатывающиеся запросы завершились
- График 2 не упал → запросы зависли (timeout)

#### Исходящие запросы (Outbound RED)

**Критически важно**: Мониторить ВСЕ исходящие запросы!

```python
class DatabaseClient:
    def __init__(self, db_connection):
        self.db = db_connection
    
    def query(self, sql, params=None):
        request_id = generate_id()
        start_time = time.time()
        
        # 1. Регистрируем попытку запроса
        metrics.record("db_requests_sent", 1, labels={
            "operation": self._get_operation(sql),
            "table": self._get_table(sql)
        })
        
        # Добавляем в трекер
        active_db_requests[request_id] = {
            "start_time": start_time,
            "query": sql
        }
        
        try:
            result = self.db.execute(sql, params, timeout=5.0)
            
            # Успех!
            duration = time.time() - start_time
            active_db_requests.pop(request_id)
            
            metrics.record("db_requests_success", 1)
            metrics.record("db_request_duration", duration)
            
            # Специальная метрика для пустых результатов
            if not result:
                metrics.record("db_requests_empty_result", 1)
            
            return result
            
        except TimeoutError:
            active_db_requests.pop(request_id)
            metrics.record("db_requests_timeout", 1)
            raise
            
        except DatabaseError as e:
            active_db_requests.pop(request_id)
            metrics.record("db_requests_error", 1, labels={
                "error_type": type(e).__name__
            })
            raise
```

**Метрики для БД**:
- `db_requests_sent` — сколько отправили
- `db_requests_success` — сколько успешно вернулись
- `db_requests_error` — сколько вернулись с ошибкой
- `db_requests_timeout` — сколько не ответили вовремя
- `db_requests_empty_result` — сколько вернули пустой результат (может быть нормой или проблемой)
- `db_request_duration` — распределение времени ответа

**Дашборд покажет**:
```
БД "users_db":
  Sent:    1000 req/s
  Success: 0 req/s     ⚠️ Проблема!
  Timeout: 1000 req/s  🚨 БД не отвечает!
  
БД "cache_redis":
  Sent:    5000 req/s
  Success: 4500 req/s
  Empty:   500 req/s   ← Нормально для кеша
```

**Сэкономленное время**: С таким мониторингом проблему нашли бы за **2 минуты** вместо 70.

### Философия алертирования

#### Главное правило
**Суть алертирования**: Привлечь внимание инженеров к **срочным и важным** событиям.
**Матрица Эйзенхауэра**:

|                | **Важное**           | **Неважное**        |
|----------------|----------------------|---------------------|
| **Срочное**    | 🔴 **АЛЕРТ**         | ⚠️ Можно подождать |
| **Несрочное**  | 📋 Задача в трекер   | 🗑️ Игнорируем      |

#### Что алертируем?
**Класс 1: Сбой (сейчас)**
```
❌ Сервис не работает
❌ Error rate > 5%
❌ p95 latency > 1s (SLA нарушен)
❌ База данных недоступна
```

**Класс 2: Скоро сбой (через минуты/часы)**
```
⚠️ Осталось 5% места на диске
⚠️ Memory usage > 90%
⚠️ CPU sustained > 85% за 10 минут
⚠️ Consumer lag растет (Kafka)
```

**НЕ алертируем**:
```
ℹ️ CPU spike на 2 секунды (несрочно, неважно)
ℹ️ Единичная ошибка (error rate < 1%)
ℹ️ Медленный запрос от одного пользователя
ℹ️ Интересные паттерны в логах (для аналитики, не для алертов)
```

#### Реальная статистика

**Диапазон алертов в компании лектора (за квартал)**:
- Хорошая команда: **20 алертов**
- Плохая команда: **2000 алертов**

**Проблема 2000 алертов**:
- Невозможно читать все
- Игнорирование алертов
- "Мобильный телефон перегревается" (из-за постоянных SMS)
- Выгорание инженеров

**Здоровая ситуация**:
```
Еженедельная встреча команды:
"На этой неделе было 3 алерта:
 1. Диск заполнился на prod-db-1 → почистили логи
 2. Spike в error rate → откатили деплой
 3. Redis недоступен 5 минут → проблема с сетью, восстановилось само"
```

**Каждый алерт** → разбор и улучшение.

### Организация дежурств
#### Антипаттерн: Групповой чат
**Плохо**:
```
[#ops-alerts]
🚨 Production DB high CPU!
🚨 Service API error rate 10%!
🚨 Disk space critical!

[Все видят, никто не реагирует]
```

**Проблема**: Диффузия ответственности.
- "Кто-нибудь среагирует"
- "Наверное, не я"
- "Уже кто-то чинит, не буду мешать"

#### Правильно: Конкретный адресат
**Структура дежурств**:
```
Неделя 1: Иван (основной), Мария (запасной)
Неделя 2: Мария (основной), Петр (запасной)
Неделя 3: Петр (основной), Иван (запасной)
```

**Алерт идет**: Ивану (основной дежурный) → понятно, что **только он** отвечает.
#### Антипаттерн: Всегда дежурит самый опытный

**Проблема**:
```
Дежурства:
Иван (Senior):  ████████████████████ 90% времени
Мария (Middle): ██                   5% времени
Петр (Junior):  ██                   5% времени
```

**Последствия**:
- Иван становится еще опытнее (single point of knowledge)
- Мария и Петр не растут
- Иван уходит в отпуск → команда беспомощна
- Иван увольняется → катастрофа

**Правильно**: Равномерное распределение или смещение в сторону **менее опытных**.

```
Дежурства:
Иван (Senior):  ████              25% времени
Мария (Middle): ████████          40% времени
Петр (Junior):  ██████            35% времени
  ↑
  └─ Больше опыта получают те, кому это нужнее
```

**Принцип**: 
> Если самый опытный всегда дежурит, опыт остальных **никогда не вырастет**.

**Применимо и к задачам**: Сложные задачи должны делать менее опытные (с поддержкой).

### Механика доставки алертов

#### Проблема: Люди спят

**Реальность**: Инженеры спят ночью (обычно 😅).

**Единственный способ разбудить**: Звонок.

**Почему SMS/push недостаточно**:
- Антиспам блокирует
- Телефон в беззвучном режиме
- Человек спит крепко

#### Решение: Звонок + подтверждение

**Алгоритм**:
```
1. Алерт сработал
2. Звонок дежурному (мобильный телефон)
3. Повторный звонок через 1 минуту (если не ответил)
4. Повторный звонок через 2 минуты
5. Если не ответил → эскалация
```

**Подтверждение приема алерта**:
```
Дежурный проснулся → 
  Заходит в систему → 
    Нажимает "Принял алерт" →
      Звонки прекращаются
```

**Почему это важно**:
- Человек может быть в сознании, но недееспособен (везут в скорую)
- Человек может сбросить звонок во сне
- Нужна явная фиксация: "Я в деле"

#### Антипаттерн: Блокировка номера
**Реальный кейс**:
```
1. Инженер настроил алерты
2. Начали звонить ночью
3. Инженер не хотел разбираться сразу
4. "Потом доделаю, а пока добавлю в blacklist"
5. Забыл убрать из blacklist
6. Прошел месяц → алерты пропускаются
```

**Решение**: Мобильное приложение (нельзя заблокировать).
#### Повторы звонков
**Проблема**: Люди могут сбросить звонок во сне.
**Решение**: Повторять звонок до подтверждения.

```python
def alert_oncall(alert):
    oncall = get_current_oncall()
    
    for attempt in range(5):  # До 5 попыток
        call(oncall.phone)
        
        # Ждем подтверждения 2 минуты
        acknowledged = wait_for_ack(timeout=120)
        
        if acknowledged:
            return True  # Дежурный принял
        
        # Не принял → звоним снова
        time.sleep(60)  # Пауза 1 минута
    
    # После 5 попыток → эскалация
    escalate(alert)
```

### Эскалация

#### Что делать, если дежурный не ответил?

**Причины**:
- Спит слишком крепко
- Телефон разрядился
- Нажал "Я не дееспособен"
- Реальная проблема (скорая помощь, аппендицит)

**Схема эскалации** (пример из компании лектора):

```
Уровень 1: Основной дежурный     [10 минут]
           ↓ (не ответил)
Уровень 2: Запасной дежурный     [10 минут]
           ↓ (не ответил)
Уровень 3: Вся команда           [10 минут]
           (звонок ВСЕМ одновременно)
           ↓ (никто не ответил)
Уровень 4: Руководитель команды  [10 минут]
           ↓ (не ответил)
Уровень 5: 24/7 инженеры         [∞]
           (дежурят круглосуточно,
            вызывают нужных людей,
            ищут соседние команды)
```

**Кнопка "Я не дееспособен"**:
```
[Принял алерт]  [Не дееспособен]
                      ↓
             Сразу эскалация на уровень 2
             (не ждем 10 минут)
```

**Автоматизация эскалации**:
```yaml
escalation:
  - level: 1
    target: primary_oncall
    timeout: 10m
    
  - level: 2
    target: secondary_oncall
    timeout: 10m
    
  - level: 3
    target: team_group
    mode: simultaneous  # Всем сразу
    timeout: 10m
    
  - level: 4
    target: team_lead
    timeout: 10m
    
  - level: 5
    target: global_ops
    mode: always_available
```

### Подавление шторма алертов (Alert Storm Suppression)

#### Проблема: Каскадный сбой

**Сценарий**:
```
База данных упала
  ↓
Все сервисы не могут подключиться
  ↓
1000 алертов за минуту:
  - service-api: DB connection failed
  - service-payment: DB connection failed
  - service-user: DB connection failed
  - service-order: DB connection failed
  ...×1000
```

**Инженер**: Не может принять 1000 алертов физически.

#### Решение 1: Задержка и группировка

**Алгоритм**:
```python
def send_alerts(alerts):
    """Smart alert delivery"""
    
    # Задерживаем на 1 минуту
    time.sleep(60)
    
    # Группируем похожие алерты
    grouped = group_similar_alerts(alerts)
    
    if len(grouped) > 10:
        # Много алертов → отправляем сводку
        send_single_alert(
            f"🚨 Alert storm: {len(grouped)} alerts\n"
            f"Top issues:\n"
            f"  - DB connection failed: 500 alerts\n"
            f"  - High latency: 300 alerts\n"
            f"  - Error rate: 200 alerts\n"
            f"\n"
            f"Full list: https://alerts.example.com/storm/abc123"
        )
    else:
        # Немного → отправляем как обычно
        for alert in grouped:
            send_alert(alert)
```

#### Решение 2: Кнопка "Принять все"

**UI**:
```
🚨 Active Alerts (47)

[Acknowledge All] [Suppress for 1h]

- DB connection failed (service-api)
- DB connection failed (service-payment)
- DB connection failed (service-user)
...
```

#### Решение 3: Корреляция

**Умная система**:
```
Алерты:
1. ❌ Database prod-db-1 down
2. ⚠️ Service-api can't connect to DB
3. ⚠️ Service-payment can't connect to DB
4. ⚠️ Service-order can't connect to DB

Корреляция:
  Root cause: Database prod-db-1 down
  Affected: service-api, service-payment, service-order
  
Отправляем:
  "🚨 Database prod-db-1 down → 3 services affected"
  (один алерт вместо 4)
```

### Дополнительные практики

#### Алерт "Восстановлено"

**Спорная практика**: Отправлять второе сообщение, когда проблема решена.

```
Алерт 1: 🚨 Database connection failed
Алерт 2: ✅ Database connection restored
```

**Мнение лектора**: Не очень нужно, проверить можно по дашбордам.

**На практике**: Инженеры любят такие уведомления (психологическое облегчение).

**Сложность реализации**: Гораздо сложнее, чем кажется.
- Как определить "восстановлено"?
- Кратковременное восстановление → снова упало?
- Частичное восстановление (50% реплик)?

#### Ссылки в алертах

**Best practice**: Сразу давать контекст.

```
🚨 High error rate in service-api

Current: 15% errors (threshold: 5%)

📊 Dashboard: https://grafana.../service-api
📈 Error graph: https://grafana.../errors
📋 Runbook: https://wiki.../service-api-errors
🔍 Recent logs: https://kibana.../...
```

**Графики в сообщении** (современные мессенджеры):
```
🚨 CPU usage critical

[Embedded graph showing spike]

Current: 95%
Previous hour average: 45%
```

### Несрочные, но важные задачи

#### Проблема

**Пример**:
```
Заполнено 70% диска
По стандарту: должно быть свободно ≥50%
По текущему темпу: через 2 недели достигнем лимита
```

**Характеристика**:
- ✅ Важно (нарушение стандарта)
- ⚠️ Не срочно (2 недели есть)

**Не нужен алерт** (не будить ночью), но **нужно действие**.

#### Решение: Автоматическое создание задач

**Система**:
```python
def check_disk_usage():
    usage = get_disk_usage()
    
    if usage > 0.7:  # 70%
        # Не алерт, а задача в Jira/Linear
        task = create_task(
            title="Disk usage >70% on prod-db-1",
            description=f"Current: {usage*100}%\n"
                        f"Projected full in: 14 days",
            priority="High",
            assignee=random_team_member(),  # Распределяем
            labels=["operations", "maintenance"]
        )
        
        # Проверка дубликатов
        if task.already_exists():
            # Не создаем повторно
            # Можем добавить комментарий раз в сутки
            if task.last_comment > 24h_ago:
                task.add_comment("Disk usage still >70%")
        
        return task
```

**Эскалация задачи**:
```
День 0:  Задача создана (Priority: Medium)
День 7:  Приоритет → High (если не решена)
День 10: Приоритет → Critical
День 12: Задача → Алерт (срочно исправить!)
```

**Преимущества**:
- Не триггерим людей ночью
- Равномерно распределяем работу
- Автоматическая эскалация при игнорировании

### Термины: Alert vs Page

**Alert** (широкое понятие):
- Триггер (условие срабатывания)
- Шаблон сообщения
- Правила доставки
- Вся система в целом

**Page** (конкретное событие):
- Факт доставки конкретного сообщения
- "Мне пришел page в 3 утра"
- Происходит от "pager" (пейджеры до сотовых телефонов)

**Пример**:
```
Alert:   "CPU > 80% за 5 минут"
         ↓ (сработал)
Page:    Звонок + SMS Ивану в 03:15
         "🚨 CPU 85% on prod-server-1"
```

### Культура дежурств

#### Правило: Дежурный всегда доступен

**В дни дежурства**:
- ❌ Не назначать внешние встречи
- ❌ Не проводить собеседования
- ❌ Не ехать в места без связи
- ✅ Всегда иметь при себе телефон и ноутбук
- ✅ Прерывать любую деятельность при алерте

**На встречах**:
```
Коллеги знают: дежурный может внезапно выбежать
→ Это нормально
→ Никто не обижается
```

**Внешние встречи**:
```
"Извините, я дежурный на этой неделе,
 могу внезапно уйти, если что-то сломается"
```

#### Реальные истории

**Новогодняя ночь**:
```
0:00 - Вся компания празднует
0:15 - Критический сбой
0:20 - Единственный трезвый: один администратор
       (сидит под елкой с ноутбуком)
       Пытается чинить в одиночку
6:00 - Разбор: "Что я вообще там наделал ночью?"
```

**Урок**: На праздники нужны **специальные дежурства** или внешний **NOC (24/7)**.

**Своровавшие серверы**:
```
Руководитель: "Достань 2 сервера откуда-нибудь"
Инженер: Взял из соседней команды
         (это были backup-серверы!)
         
Соседи: Автоматический failover не работает
        Разбираются всю ночь
        Оказывается: серверов нет физически 🤦
```

## Выводы и инсайты

- **RED метрики на ВХОДЕ**: Регистрируйте запросы СРАЗУ при получении, иначе пропустите сбои
- **RED метрики на ВЫХОДЕ**: Мониторьте ВСЕ исходящие запросы (БД, API, очереди) — именно там чаще всего проблемы
- **70 минут → 2 минуты**: Правильный мониторинг сокращает время поиска проблемы в **35 раз**
- **Алерты только для срочного и важного**: 20 алертов/квартал — здорово, 2000 — катастрофа
- **Конкретный адресат**: Групповой чат убивает ответственность
- **Дежурят менее опытные**: Единственный способ вырастить команду
- **Звонок + подтверждение + эскалация**: Единственный надежный способ доставки
- **Подавление шторма**: Система должна группировать алерты при каскадных сбоях
- **Задачи для несрочного**: Не все важное должно быть алертом
- **При каждом долгом поиске**: "Какого мониторинга мне не хватило?"
- **Культура дежурств**: Дежурный всегда прерывает любую деятельность при алерте
- **Page = призыв к действию**: Исторический термин от пейджеров, означает факт доставки
- **Не виним людей за сон**: Инженеры должны спать, система должна их будить корректно
- **Автоматы за посещаемость**: 100% посещаемость лекций → автомат (по усмотрению лектора)
