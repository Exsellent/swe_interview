## О чем лекция
Лекция посвящена проектированию систем с учётом ненадёжности зависимостей: как обрабатывать отказы, правильно использовать таймауты и повторы, избегать самоDDoS и восстанавливаться после перегрузок.

## Ключевые вопросы
- Что делать, если зависимость не отвечает?
- Как правильно повторять запросы, не убив систему?
- Что такое дедупликация и зачем она нужна?
- Какие ресурсы могут исчерпаться и как к этому готовиться?
- Как система должна вести себя при 10x нагрузке?

## Основная часть

### Работа с зависимостями: 4 варианта отказа

**Что может пойти не так с зависимостью**:

```
1. ❌ Не можем подключиться (connection refused)
2. ❌ Подключились, но нет ответа (timeout)
3. ❌ Ответ с ошибкой (500, 503, etc.)
4. ⚠️ Отвечает, но медленнее обычного
```

**Самый сложный случай**: №4 — медленные ответы

#### Почему медленные ответы опасны

**Проблема**: Один медленный узел съедает все соединения

```
[Ваш сервис] → [Кластер БД]
                   ├─ Нода 1 (быстрая)
                   ├─ Нода 2 (быстрая)
                   └─ Нода 3 (медленная) ← Проблема

Что происходит:
1. Запросы к ноде 3 висят долго
2. Connection pool заполняется ожидающими
3. Новые запросы не могут получить соединение
4. Вся система встаёт
```

**Реальные причины медленной работы**:
- План запроса перестроился (БД выбрала плохой индекс)
- Запустился vacuum/garbage collection
- Один процесс начал активно писать на диск
- Сеть деградировала (packet loss)

**Решение**: Circuit breaker (переключатель) + таймауты

### Таймауты — критически важны

**Проблема**: Таймаут по умолчанию вам не подходит

**Типы таймаутов** (по Марине Културной):
- Timeout на установку TCP-соединения
- Timeout на получение первого байта
- Timeout на полный ответ
- Timeout на чтение из сокета

**Принцип**: У вас должны быть **явно выставленные** таймауты на каждый тип операции.

**Пример настройки**:
```python
# ❌ Плохо: таймаут по умолчанию (может быть бесконечным)
response = requests.get('http://api.example.com/data')

# ✅ Хорошо: явные таймауты
response = requests.get(
    'http://api.example.com/data',
    timeout=(3.0, 10.0)  # (connect timeout, read timeout)
)
```

### Circuit Breakers (Переключатели)

**Идея**: Временно отключать проблемную зависимость

**Состояния**:
```
CLOSED (нормальная работа):
  → Запросы идут к зависимости
  → Если ошибки > threshold → OPEN

OPEN (зависимость отключена):
  → Запросы НЕ идут (сразу возвращаем ошибку)
  → Периодически пробуем (раз в минуту) → HALF-OPEN

HALF-OPEN (проверка восстановления):
  → Пробный запрос
  → Успех → CLOSED
  → Ошибка → OPEN
```

**Зачем нужно**:
- Не DDoS'ить упавшую зависимость попытками подключения
- Дать ей время восстановиться
- Быстро восстановить работу при подъёме

### Реальный кейс: Банкоматы и самоDDoS

**Что произошло**:
```
1. Сервер банкоматов моргнул (кратковременный сбой)
2. Все банкоматы потеряли соединение
3. Все одновременно пытаются переподключиться
4. Сервер не выдерживает массового подключения
5. Сервер падает снова
6. Бесконечный цикл: поднимается → падает
```

**Проблема**: Подключение дороже, чем обработка запросов

**Как починили** (повезло с архитектурой):
```
1. Банкоматы были на разных операторах связи
2. Отключили сеть для 2/3 операторов
3. Оставили только 1/3 банкоматов
4. Сервер выдержал их подключение
5. Постепенно добавили остальных (по 1/3)
```

**Вывод**: Переподключение должно быть **с джиттером** (случайной задержкой)

## Повторы запросов (Retry Logic)

### Когда повторять полезно

**Сценарий**: Одна нода кластера упала, но есть другие

```
[Клиент] → [Load Balancer]
              ├─ Нода 1 ❌ Упала
              ├─ Нода 2 ✅
              └─ Нода 3 ✅

Первый запрос → Нода 1 → Ошибка
Повтор → Нода 2 → Успех
```

**Эффект**: Пользователь не заметит (чуть дольше, но работает)

### Когда повторять опасно

**Проблема**: Повторы создают **экспоненциальную нагрузку**

```
Нормальная нагрузка: 1000 RPS
Все запросы повторяются 1 раз: 2000 RPS (x2)
Эти повторы тоже повторяются: 4000 RPS (x4)

→ Сервер умирает от самоDDoS
```

**Каскадный эффект**:
```
[Клиент] → [Сервис A] → [Сервис B] → [Сервис C]

Если повторы на каждом уровне:
  Сервис A повторяет 2 раза → x2
  Сервис B повторяет каждый запрос 2 раза → x4
  Сервис C получает x8 нагрузки
  
→ Смерть системы
```

### Правила безопасных повторов

**Правило 1**: Максимум **2 попытки** (не больше!)

```
1 попытка → Ошибка
2 попытка (быстро, через ~1 сек) → Ошибка
→ Переключатель срабатывает, медленные повторы для восстановления
```

**Правило 2**: Джиттер (случайная задержка)

```python
# ❌ Плохо: все повторяют строго через 3 секунды
time.sleep(3)

# ✅ Хорошо: случайная задержка
import random
time.sleep(random.uniform(2, 4))
```

**Зачем**: Избежать синхронного шторма запросов при восстановлении

**Правило 3**: Повторять только **идемпотентные операции**

### Идемпотентность

**Определение**:
> Операция идемпотентна, если её можно выполнить несколько раз с тем же результатом.

**Примеры**:

```
✅ Идемпотентные:
  - GET /user/123 (чтение)
  - PUT /user/123 {"name": "Alice"} (установка значения)
  - DELETE /user/123 (повторное удаление = no-op)

❌ НЕ идемпотентные (без дедупликации):
  - POST /transfer {"amount": 500} (перевод денег)
  - POST /message {"text": "Hello"} (отправка сообщения)
  - PUT /counter/increment (увеличение счётчика)
```

**Почему это важно для повторов**:
```
Клиент отправил: "Перевести 500₽"
Сервер выполнил
Ответ потерялся в сети
Клиент повторил: "Перевести 500₽"

Без дедупликации: Переведёт 1000₽ ❌
С дедупликацией: Вернёт результат первой операции ✅
```

## Дедупликация запросов

### Как работает

**Ключ дедупликации** (Idempotency Key):

```
Запрос с ключом:
{
  "idempotency_key": "17",
  "amount": 500,
  "from": "Вася",
  "to": "Петя"
}
```

**Сервер**:
```
1. Получил запрос с ключом "17"
2. Проверяет: делали ли мы запрос с ключом "17"?
   - НЕТ → Выполняем, сохраняем ключ + результат
   - ДА → Возвращаем сохранённый результат (не выполняем повторно)
```

### Два сценария

**Сценарий 1: Два разных перевода**
```
Запрос 1: {key: "17", amount: 500} → Выполнен
Запрос 2: {key: "18", amount: 500} → Выполнен (новый ключ)

Результат: Переведено 1000₽ ✅ (так и задумано)
```

**Сценарий 2: Повтор из-за сбоя сети**
```
Запрос 1: {key: "17", amount: 500} → Выполнен
Ответ потерялся ❌
Запрос 2: {key: "17", amount: 500} → Возвращён старый результат

Результат: Переведено 500₽ ✅ (дедупликация сработала)
```

### Требования к ключу дедупликации

**Недостаточно одного ID**:
```
❌ Опасно:
  idempotency_key: "17"
  
  Злоумышленник может подобрать и повторить чужой перевод
```

**Правильно**:
```
✅ Безопасно:
  idempotency_key: "17"
  + Запрос должен быть строго идентичен
  + От того же пользователя
  + С теми же параметрами
  + Signature/HMAC для защиты
```

## Исчерпание ресурсов

### Что может исчерпаться

**Инфраструктура**:
- Диск (самое простое для мониторинга)
- Оперативная память
- CPU
- Сетевая пропускная способность

**Приложение**:
- Connection pool (соединения к БД)
- Thread pool (потоки обработки)
- Внутренние очереди (event loop в Node.js)
- Кэш (если ограничен по размеру)

**Данные**:
- 32-bit идентификаторы (~4 млрд)
- Слоты в структурах данных
- Лимиты API (rate limits)

### Реальные кейсы

#### Кейс 1: Отчёт с 4000 карт

**Проблема**:
```
Крупный клиент → 4000 карт
Запрос отчёта → Сервер пытается обработать все 4000
Для каждой карты → запрос в другой сервис
→ 4000 запросов к другому сервису
→ Тот сервис: для каждого запроса получает данные по ВСЕМ картам
→ База данных умирает
```

**Два бага одновременно**:
1. Не ограничили количество карт в отчёте (нет лимита)
2. Сервис делал O(N²) операций вместо O(N)

**Вывод**: Нужны **лимиты на максимальные значения**

#### Кейс 2: Исчерпание 32-bit ID

**Что случилось**:
```
В нескольких таблицах одновременно кончились ID
(достигли ~4 млрд)

Причина: ID расходовались с одинаковой скоростью
→ Закончились примерно одновременно
→ Серия сбоев в разных местах
```

**Решение**: Мигрировали на 64-bit ID

### Что делать с лимитами

**При проектировании**:
```
Вопросы:
1. Сколько максимум сотрудников в компании?
2. Сколько максимум товаров в корзине?
3. Сколько максимум участников в чате?
4. Сколько максимум карт у клиента?

Ответы:
- НЕ "неограниченно" ❌
- Конкретное число (1000, 10000, 1млн) ✅
```

**Тестирование**:
```
Протестировать на максимальном значении:
  - Работает ли вообще?
  - Сколько памяти/CPU потребляет?
  - Сколько времени обработка?
  
Если не работает → снизить лимит или оптимизировать
```

## Поведение при аномальной нагрузке

### Типичное тестирование (недостаточное)

```
Нагрузочное тестирование:
  Норма:   1000 RPS ✅
  x1.5:    1500 RPS ✅
  x2:      2000 RPS ✅
  x3:      3000 RPS ✅
  
→ "Держим до x3, отлично!"
```

**Проблема**: Не проверили, что будет при x10

### Что НУЖНО проверить

**Тест должен включать**:

```
1. Норма (1000 RPS)
   → Работает стабильно ✅
   
2. x2 (2000 RPS) — расчётный запас
   → Работает ✅
   
3. x10 (10000 RPS) — аномальная нагрузка
   → Что произойдёт?
   
4. Возврат к норме (1000 RPS)
   → Восстановится ли автоматически?
```

### Проблема восстановления

**Типичное поведение** (плохое):

```
Нагрузка:
  Норма → x10 → Норма
  
Что происходит:
1. x10: очереди забились, память заполнилась
2. Вернулись к норме
3. Система НЕ восстанавливается ❌
   (очереди всё ещё забиты, memory leak)
   
→ Работает только после перезапуска
```

**Правильное поведение**:

```
Нагрузка:
  Норма → x10 → x1.8 → Восстановление
  
1. x10: Rate limiting, отклоняем часть запросов
2. Очереди НЕ переполняются
3. x1.8: Постепенно обрабатываем
4. Восстановление к норме автоматически ✅
```

**Пример из Google**:
> Обязательный тест перед включением кластера поиска в продакшен: "Overload recovery test". Проверяли, что после x10 нагрузки кластер восстанавливается автоматически.

### Rate Limiting

**Что делать при x10 нагрузке**:

```
Варианты:
1. ✅ Возвращать 429 Too Many Requests (честно)
2. ✅ Обрабатывать только часть (приоритетных)
3. ❌ Пытаться обработать всё → умереть
```

**Встроенный rate limiter**:
```python
from ratelimit import limits

@limits(calls=1000, period=60)  # Максимум 1000 запросов в минуту
def handle_request():
    # Если превышен лимит → автоматически 429
    ...
```

## Ключ дедупликации на практике

### Генерация ключа

**На клиенте** (мобильное приложение):
```python
def create_transfer(amount, recipient):
    idempotency_key = str(uuid.uuid4())  # Уникальный для каждой операции
    
    try:
        response = api.post('/transfer', {
            'idempotency_key': idempotency_key,
            'amount': amount,
            'recipient': recipient
        })
        return response
    except NetworkError:
        # Повтор с ТЕМ ЖЕ ключом
        response = api.post('/transfer', {
            'idempotency_key': idempotency_key,  # Тот же!
            'amount': amount,
            'recipient': recipient
        })
        return response
```

### Обработка на сервере

```python
# Хранилище выполненных операций
completed_operations = {}  # {key: result}

def handle_transfer(request):
    key = request['idempotency_key']
    
    # Проверяем дедупликацию
    if key in completed_operations:
        # Это повтор → возвращаем старый результат
        return completed_operations[key]
    
    # Новая операция → выполняем
    result = execute_transfer(
        amount=request['amount'],
        recipient=request['recipient']
    )
    
    # Сохраняем для дедупликации
    completed_operations[key] = result
    
    return result
```

**TTL для ключей**: Не храним вечно (7-30 дней обычно достаточно)

## Мониторинг состояния системы

### Что нужно видеть

**Критичная информация**:
- Какие переключатели (circuit breakers) в каком состоянии
- Когда переключились (timestamp)
- Сколько повторов происходит сейчас
- Какие зависимости отвечают медленно
- Степень заполнения исчерпаемых ресурсов

**Пример из Google**:
> В каждом серьёзном приложении была встроенная консоль (dashкboard), где можно было зайти и увидеть состояние приложения с подсветкой (зелёное = OK, красное = проблема).

### Проблема: Как мониторить исчерпание

**Наивный подход** (неправильный):
```python
# ❌ Мониторить абсолютное значение бесполезно
current_connections = db_pool.active_count()
# 50 соединений — это много или мало?
```

**Правильный подход**:
```python
# ✅ Мониторить % заполнения
utilization = db_pool.active_count() / db_pool.max_size * 100
# 90% заполнения → скоро проблема
```

**Почему**:
- Connection pool размером 10 → 5 активных = 50% (нормально)
- Connection pool размером 100 → 5 активных = 5% (много свободных)

## Выводы и инсайты

- **Знайте ответ: "Что если зависимость не отвечает?"**: Для каждой зависимости должно быть **спроектированное** поведение, а не "наверное, не будет работать"

- **Медленные ответы опаснее отказов**: Съедают connection pool, сложно детектировать, тяжело тестировать

- **Таймауты обязательны и явные**: Таймаут по умолчанию = рулетка

- **Максимум 2 повтора**: Больше → экспоненциальный рост нагрузки и самоDDoS

- **Джиттер критичен**: Избегайте синхронных штормов при восстановлении

- **Повторы только для идемпотентных операций**: Иначе нужна дедупликация

- **Дедупликация = idempotency key**: UUID генерируется на клиенте, сервер запоминает выполненные операции

- **Circuit breakers обязательны**: Не DDoS'ить упавшие зависимости

- **Лимиты на всё**: Нет "неограниченного количества" — всё должно иметь максимум

- **Тестируйте x10 нагрузку**: И проверяйте восстановление после неё

- **Большинство приложений НЕ восстанавливается**: После перегрузки работают только после рестарта (если не спроектировали правильно)

- **Мониторьте исчерпаемое**: Диск, память, connection pool, rate limits

- **Переподключение дороже работы**: Особенно для БД — ограничивайте попытки

- **Архитектура должна быть наблюдаемой**: Состояние переключателей, повторов, зависимостей должно быть видно

- **Опциональные зависимости — реально опциональны?**: Протестируйте, что работаете без них
