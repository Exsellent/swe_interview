
## О чем лекция
Лекция посвящена практическим аспектам проектирования архитектуры: диагностике систем, обеспечению надежности и выбору между монолитной и микросервисной архитектурами.

## Ключевые вопросы
- Как правильно проектировать систему для быстрой диагностики проблем?
- Какие механизмы помогают отслеживать и решать инциденты?
- Когда выбирать монолит, а когда — микросервисы?
- Какие компромиссы существуют между разными архитектурными подходами?

## Основная часть

### Принципы проектирования надежных систем

**Сквозные идентификаторы (Trace ID)**
- Позволяют отслеживать запрос через всю систему
- Критически важны для быстрого поиска проблем в распределенных системах

**Готовность к перезапуску**
- Проектировать сервисы с учетом возможности перезапуска
- Заранее продумывать процедуры graceful shutdown и recovery

**Работа со временем**
- Логировать время в UTC или с явным указанием часового пояса
- Избегать проблем с временными зонами и переходами на летнее/зимнее время

**Влияние на пользователя**
- Всегда учитывать, как ошибки и деградация сервиса влияют на пользовательский опыт
- Проектировать систему с учетом graceful degradation

**Диагностический порт**
- Основной порт — для рабочей нагрузки
- Отдельный порт — для страницы диагностики с информацией о состоянии системы
- Диагностика работает быстрее, чем поиск по логам

**Углубленная диагностика запросов**
- В API можно предусмотреть специальный флаг для детальной записи информации о конкретном запросе
- Пример: Google использовал подобный подход для отладки поиска

### Монолит vs Микросервисы

#### Преимущества монолита

| Аспект | Преимущество |
|--------|--------------|
| **Отладка** | Легче искать проблемы и ошибки в едином кодовой базе |
| **Поддержка** | Проще обновлять и обслуживать систему |
| **Надежность** | Устойчивость к сетевым сбоям (нет межсервисных вызовов) |
| **Совместимость** | Меньше проблем с обратной совместимостью внутреннего API |
| **Производительность (Latency)** | Лучше из-за отсутствия сетевых вызовов |
| **Производительность (CPU)** | Меньше нагрузка на CPU (нет сериализации/десериализации) |
| **Архитектура** | Как правило, проще архитектура |
| **Логирование** | Меньше объем логов (логи потребляют значительные ресурсы) |

#### Преимущества микросервисов

| Аспект | Преимущество |
|--------|--------------|
| **Deployment** | Можно обновлять отдельные компоненты независимо |
| **Отказоустойчивость** | Падение одного сервиса не влияет на остальные |
| **Скорость релизов** | Можно релизить компоненты чаще и независимо |
| **Изоляция ресурсов** | Возможность гарантировать ресурсы для критичных сервисов |
| **Технологическая свобода** | Можно писать разные сервисы на разных языках программирования |
| **Организация команды** | Проще организовать работу: одна команда = один сервис |

## Выводы и инсайты

- **Диагностика — это проектирование**: Закладывайте механизмы отладки на этапе проектирования, а не после возникновения проблем
- **Нет универсального решения**: Выбор между монолитом и микросервисами зависит от контекста — размера команды, требований к масштабированию и скорости релизов
- **Производительность vs Гибкость**: Монолит выигрывает в производительности, микросервисы — в гибкости и устойчивости
- **Начинайте с монолита**: Для большинства проектов разумно начинать с монолита и переходить к микросервисам по мере роста и появления реальной необходимости
