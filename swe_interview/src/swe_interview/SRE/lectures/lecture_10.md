## Основная часть

### Цели мониторинга

Мониторинг — это не просто сбор метрик. У него есть **конкретные цели**, и к каждой нужен свой подход.

#### 1. Детектирование сбоев

**Задача**: Обнаружить проблемы до того, как пользователи пожалуются.

**Что контролируем**:
- **SLA (Service Level Agreement)** — соблюдаем ли договоренности об уровне сервиса
- **SLO (Service Level Objectives)** — целевые показатели
- **SLI (Service Level Indicators)** — метрики для измерения

**Пример**:
```
SLA: 99.9% uptime
SLO: 95% запросов быстрее 200ms
SLI: Latency p95, Error Rate

Alert: если Error Rate > 1% в течение 5 минут → инцидент
```

**Сложность**: Нужно по данным **автоматически** решать, соблюдаем ли мы SLA.

#### 2. Поиск причин сбоев при устранении

**Задача**: Быстро понять, **что именно** сломалось.

**Инструменты**:
- **Информативные дашборды** — графики состояния систем
- **Корреляция метрик** — сопоставление изменений в разных подсистемах
- **Drill-down** — углубление от общих метрик к деталям

**Пример дашборда**:
```
[CPU Usage] [Memory Usage] [Disk I/O]
[Request Rate] [Error Rate] [Latency]
[Database Connections] [Cache Hit Rate]
```

При инциденте:
1. Смотрим общие метрики → Error Rate вырос
2. Drill-down → ошибки в Payment Service
3. Drill-down → Database connection pool exhausted
4. **Причина найдена**: DB перегружена

#### 3. Сбор информации для отладки

**Задача**: Детальные данные для анализа конкретных проблем.

**Инструменты**:
- **Детальные логи** с контекстом
- **Трейсы** (distributed tracing)
- **Метрики с высокой детализацией**

**Пример**:
```
Request ID: abc-123-def
Timestamp: 2024-01-15 14:23:45
Latency breakdown:
  - Auth: 15ms
  - DB Query: 450ms ⚠️
  - External API: 30ms
  - Serialization: 5ms
Total: 500ms
```

#### 4. Выявление трендов

**Задача**: Прогнозирование и планирование ресурсов.

**Примеры**:
- **Рост использования CPU** → прогноз закупки железа
- **Увеличение расходов на инфраструктуру** → оптимизация затрат
- **Рост количества пользователей** → планирование масштабирования
- **Деградация производительности** → технический долг накапливается

**Графики трендов**:
```
CPU Usage (30 дней):
50% → 55% → 60% → 65% → ...

Прогноз: через 2 месяца достигнем 80% → нужно масштабироваться
```

### Метрики: Серия значений во времени

**Определение**: Метрика — это числовое значение, измеренное в определенный момент времени.

```
Пример метрики:
http_requests_total{service="api", method="GET", status="200"}

Временной ряд:
2024-01-15 14:00:00 → 1000
2024-01-15 14:01:00 → 1050
2024-01-15 14:02:00 → 1120
2024-01-15 14:03:00 → 1095
```

---

### Типы метрик

#### 1. Counter (Счетчик)

**Особенность**: Значение **только растет**, никогда не убывает (или сбрасывается в 0).

**Примеры**:
- Общее количество HTTP-запросов
- Количество обработанных сообщений
- Количество ошибок (всего)

```python
http_requests_total = 0

@app.route('/api/users')
def get_users():
    http_requests_total += 1  # Только инкремент!
    return users
```

**Почему это круто**:
- ✅ Легко считать **скорость изменения** (rate)
- ✅ Монотонный рост — просто анализировать
- ✅ Restart-safe: если приложение перезапустилось, видим сброс в 0

**Запросы к Counter**:
```promql
# Количество запросов в секунду за последние 5 минут
rate(http_requests_total[5m])

# Общий прирост за час
increase(http_requests_total[1h])
```

#### 2. Gauge (Измеритель)

**Особенность**: Значение может **расти и убывать**.

**Примеры**:
- Текущее использование CPU (%)
- Свободное место на диске (GB)
- Количество активных соединений
- Температура сервера

```python
disk_free_bytes = 50_000_000_000  # 50 GB

# Может изменяться в любую сторону
disk_free_bytes -= 1_000_000  # Файл записан
disk_free_bytes += 5_000_000  # Файл удален
```

**Когда использовать**:
- Текущее состояние системы
- Значения, которые изменяются непредсказуемо

**Важно**: Если метрика **не сильно изменяется**, Gauge идеален (например, свободное место на диске изменяется плавно).

#### 3. Histogram (Гистограмма с бакетами)

**Особенность**: Распределение значений по **диапазонам** (buckets).

**Пример**: Latency запросов
```
Buckets:
≤ 5ms:   1000 запросов
≤ 10ms:  2500 запросов
≤ 20ms:  4000 запросов
≤ 50ms:  4800 запросов
≤ 100ms: 5000 запросов
```

**Визуализация**:
```
    │     ████
    │     ████
4000│ ████████
    │ ████████
2000│ ████████
    │ ████████ ██
    └─────────────────
      5  10 20 50 100 (ms)
```

**Преимущества**:
- ✅ Видим **распределение** значений
- ✅ Можем считать **процентили** (percentiles)
- ✅ Агрегируются хорошо (можно суммировать бакеты)

**Запросы**:
```promql
# 95-й процентиль latency
histogram_quantile(0.95, rate(http_request_duration_bucket[5m]))
```

#### 4. Histogram (Quantiles/Percentiles)

**Особенность**: Хранение **предвычисленных процентилей**.

**Пример**:
```
http_request_duration_seconds{quantile="0.5"}  = 0.015  # медиана
http_request_duration_seconds{quantile="0.95"} = 0.045  # 95%
http_request_duration_seconds{quantile="0.99"} = 0.120  # 99%
```

**Разница с бакетами**:
| Бакеты | Квантили |
|--------|----------|
| Можно агрегировать между серверами | Нельзя агрегировать корректно |
| Процентили считаются приблизительно | Процентили точные |
| Гибкие запросы (любой percentile) | Фиксированные percentiles |

**Когда использовать квантили**:
- Одиночный сервер (не нужна агрегация)
- Известны заранее нужные percentiles

**Best practice**: Используйте **бакеты** для распределенных систем.

### Методы сбора метрик

#### Push (Заталкивание)

**Принцип**: Приложение **само отправляет** метрики в систему мониторинга.

```
[Приложение] --push--> [Pushgateway] --> [Prometheus]
     │                       │
     │                       ├─ метрики от App1
     └─ активная отправка    ├─ метрики от App2
                             └─ метрики от App3
```

**Примеры систем**: StatsD, Graphite, CloudWatch

**Преимущества**:
- ✅ Работает для **короткоживущих процессов** (batch jobs, lambda)
- ✅ Приложение контролирует, когда отправлять
- ✅ Проще для приложений за NAT/Firewall

**Недостатки**:
- ⚠️ Система мониторинга не знает, **жив ли сервис** (если сервис упал, метрики просто перестают приходить)
- ⚠️ Дополнительная нагрузка на приложение
- ⚠️ Нужна промежуточная система (pushgateway)

#### Pull (Вытягивание)

**Принцип**: Система мониторинга **сама запрашивает** метрики у приложения.

```
[Prometheus] --pull--> [App1:9090/metrics]
     │                 [App2:9090/metrics]
     │                 [App3:9090/metrics]
     │
     └─ периодически (каждые 15s)
```

**Примеры систем**: Prometheus

**Преимущества**:
- ✅ Система мониторинга **знает о состоянии сервиса** (если не отвечает → dead)
- ✅ Легче контролировать нагрузку (rate limiting на стороне Prometheus)
- ✅ Service discovery (автоматическое обнаружение новых инстансов)

**Недостатки**:
- ⚠️ Может создать **DDoS** на приложение, если scrape interval слишком частый
- ⚠️ Не подходит для короткоживущих процессов
- ⚠️ Проблемы с файрволами (приложение должно быть доступно для Prometheus)

**Защита от DDoS**:
```yaml
# prometheus.yml
scrape_configs:
  - job_name: 'my-app'
    scrape_interval: 15s     # Не чаще 15 секунд
    scrape_timeout: 10s      # Таймаут запроса
```

### Работа с высокочастотными метриками

**Проблема**: Метрики, которые изменяются очень часто (CPU, пулы соединений) могут создать огромную нагрузку.

#### Пример 1: Использование CPU

**Неправильный подход** (Gauge текущего значения):
```python
cpu_usage_percent = get_current_cpu()  # 45.3%
```

**Проблема**: Значение постоянно скачет, сложно анализировать.

**Правильный подход** (Counter суммарного времени):
```python
# Метрики:
cpu_seconds_total = 12345.67  # Суммарное время работы CPU (секунды)
cpu_seconds_idle  = 5678.90   # Суммарное время простоя CPU (секунды)
```

**Как считать использование**:
```promql
# % использования CPU за последнюю минуту
rate(cpu_seconds_total[1m]) / 
  (rate(cpu_seconds_total[1m]) + rate(cpu_seconds_idle[1m])) * 100
```

**Почему это лучше**:
- ✅ Counter монотонно растет (проще работать)
- ✅ Можем считать **среднее за период**
- ✅ Агрегируется корректно

#### Пример 2: Пулы соединений

**Метрики**:
```python
# Для каждого пула соединений
connection_pool_active_time_seconds{pool="db_main"}  # Суммарное время активных соединений
connection_pool_idle_time_seconds{pool="db_main"}    # Суммарное время простоя
connection_pool_size{pool="db_main"}                 # Размер пула (Gauge)
```

**Расчет среднего использования**:
```promql
# Средний % занятых соединений за последние 5 минут
rate(connection_pool_active_time_seconds[5m]) / 
  connection_pool_size * 100
```

### Проблемы масштабирования Prometheus

#### Проблема: Кардинальность (Cardinality)

**Что это**: Количество уникальных комбинаций меток (labels).

**Пример**:
```python
http_requests_total{
    service="api",
    method="GET",
    endpoint="/users",
    status="200",
    user_id="12345"  # ⚠️ Проблема!
}
```

**Расчет кардинальности**:
```
services (10) × methods (7) × endpoints (100) × statuses (5) × user_ids (1_000_000)
= 35 миллиардов временных рядов! 💥
```

**Prometheus умрет** от такого количества метрик.

#### Решение 1: Ограничить кардинальность

**Плохо**:
```python
# ❌ user_id в метках — миллионы уникальных значений
http_requests_total{user_id="12345"}
```

**Хорошо**:
```python
# ✅ Только общие категории
http_requests_total{service="api", endpoint="/users", status="200"}
```

**Правило**: Используйте метки только для **ограниченного набора значений**.

#### Решение 2: Агрегация по времени

**Проблема**: Хранение метрик с высокой частотой (каждую секунду) долго не масштабируется.

**Решение**: Downsampling (снижение частоты)
```
Raw data (15s):     [хранить 7 дней]
Aggregated (1m):    [хранить 30 дней]
Aggregated (1h):    [хранить 1 год]
Aggregated (1d):    [хранить вечно]
```

**Инструменты**:
- Prometheus + Thanos (долгосрочное хранение)
- VictoriaMetrics (более эффективное хранение)
- M3DB (распределенное хранение временных рядов)

#### Решение 3: Распределение метрик

**Федерация Prometheus**:
```
[Prometheus Global]
    ↑     ↑     ↑
    │     │     │
[Prom1] [Prom2] [Prom3]  ← Локальные инстансы
  ↑       ↑       ↑
Apps1   Apps2   Apps3
```

**Шардирование**:
- Каждый Prometheus отвечает за свою группу сервисов
- Глобальный Prometheus агрегирует только важные метрики
- Детальные метрики — локально

## Выводы и инсайты

- **Мониторинг ≠ Логи**: Метрики для нахождения проблем, логи для понимания причин
- **Разные цели → Разные подходы**:
  - SLA мониторинг → алерты на нарушения
  - Debugging → детальные метрики и трейсы
  - Тренды → долгосрочное хранение агрегированных данных
- **Counter > Gauge для большинства случаев**: Counter монотонно растет, проще анализировать и агрегировать
- **Высокочастотные метрики → Суммарное время**: Трекайте не текущее значение, а накопленное время
- **Pull лучше Push для долгоживущих сервисов**: Prometheus знает, жив ли сервис
- **Push для batch jobs**: Короткоживущие процессы не могут ждать scrape
- **Кардинальность убьет Prometheus**: Избегайте высококардинальных меток (user_id, request_id, timestamp)
- **Агрегируйте по времени**: Храните детальные данные недолго, агрегированные — дольше
- **Histogram buckets > quantiles**: Для распределенных систем предпочитайте бакеты
- **Мониторинг должен быть дешевле, чем система**: Если мониторинг потребляет 30% ресурсов — что-то не так
- **SLO-based alerting**: Алертите на нарушения SLO, а не на arbitrary thresholds
- **Начните просто**: Базовые метрики (latency, error rate, throughput) покрывают 80% проблем
- **Следите за трендами**: Производительность деградирует постепенно, замечайте это заранее
