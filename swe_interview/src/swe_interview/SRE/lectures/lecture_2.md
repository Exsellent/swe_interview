## О чем лекция
Лекция посвящена разложению понятия надёжности на составляющие, пониманию того, что влияет на скорость восстановления, проблемам асинхронных архитектур и базовым принципам работы с надёжностью систем.

## Ключевые вопросы
- Из чего состоит надёжность системы?
- Что значит "быстро починить" и какие этапы это включает?
- Почему IT-системы принципиально неустойчивы?
- Какие проблемы создают асинхронные очереди?
- Какие принципы должен знать каждый, кто работает с надёжностью?

## Декомпозиция надёжности
### Формула надёжности
**Надёжная система** = Редко ломается + Быстро чинится

```
Надёжность
    ├─ 1. Редко ломается
    │     ├─ Не для всех пользователей
    │     └─ Не весь функционал
    │
    └─ 2. Быстро чинится
          ├─ 2.1. Быстро обнаружить
          ├─ 2.2. Быстро найти причину
          ├─ 2.3. Быстро починить
          └─ 2.4. Устранить последствия
```

---
## 1. Редко ломается

### Что влияет на частоту поломок
**Факторы** (из прошлой лекции):
- Архитектура
- Качество кода
- Настройки
- Выбор железа
- Процессы работы команды
- Квалификация людей

**Важный нюанс**: "Редко" и "не полностью"

**Не полностью значит**:
```
✅ Работает для 99.9% пользователей
✅ Отказал только веб, но мобильные приложения работают
✅ Сломался один эндпоинт из 100
```

**Реальный пример**:
> Недавно не работал веб-интерфейс, но мобильные приложения работали. Обращений в колл-центр почти не было — мобильные приложения спасли ситуацию.

---

## 2. Быстро чинится

### 2.1. Быстро обнаружить

**Зачем**: Чем раньше узнали о проблеме, тем раньше начали чинить.

**Инструменты**:
- Мониторинг
- Алертинг
- Обращения пользователей в колл-центр

**Антипример**:
> Проблему обнаружили через 2 часа → начали чинить через 2 часа → время недоступности увеличилось на 2 часа.

### 2.2. Быстро найти причину

**Важно**: Не всегда нужно найти причину, чтобы починить!

**Универсальный способ** (без знания причины):
- Перезапуск (Simple Reboot Engineer)
- Переключение на реплику
- Увод трафика с проблемных нод

**Но**: После быстрой починки ОБЯЗАТЕЛЬНО нужно найти причину, чтобы предотвратить повторение.

### 2.3. Быстро починить

**Временная починка** ≠ **Устранение корневой причины**

**Примеры временной починки**:
```
Сломалась нода → Увели трафик → "Починили" (работает)
НО: Нода всё ещё сломана → Нужно разобраться почему
```

**Ошибка**: "Увели трафик и думаем, что всё хорошо" 🚨

### 2.4. Устранить последствия

**Проблема**: Инженеры часто забывают про этот этап.

**Пример: Очередь SMS-сообщений**

#### Архитектура системы SMS

```
[Различные сервисы] → [Kafka очередь] → [SMS-отправитель] → [Мобильные сети]
```

**Типы SMS**:
1. **Уведомления о транзакциях** (несрочные)
2. **Одноразовые пароли** (срочные, TTL ~5 минут)

#### Проблемный сценарий

**Что произошло**:
```
1. SMS-отправитель упал на 2 часа
2. В Kafka накопилось 2 миллиона сообщений
3. Систему восстановили
4. Начали отправлять сообщения из очереди
```

**Проблемы**:

**Проблема 1**: Разгребание очереди занимает ещё 30-60 минут
```
Система "восстановлена", но:
- Старые SMS всё ещё отправляются
- Новые одноразовые пароли отправятся только через час
- Для пользователя система НЕ работает
```

**Проблема 2**: Одноразовые пароли просрочены
```
Пользователь запросил пароль 2 часа назад
Получит его сейчас (спустя 2.5 часа)
Пароль уже не валиден → бесполезен
```

**Вывод**: 
> Восстановление работоспособности ≠ Устранение последствий

#### Решения для SMS-очереди

**Вариант 1**: Очистка просроченных сообщений
```python
# При восстановлении: удалить все одноразовые пароли старше 5 минут
def cleanup_expired_otp():
    messages = kafka.read_all()
    for msg in messages:
        if msg.type == "OTP" and msg.age > 300:  # 5 минут
            kafka.delete(msg)
```

**Вариант 2**: Две отдельные очереди (лучше!)
```
[Kafka: Уведомления] ← Длинная очередь (хранить долго)
        ↓
[SMS-отправитель]

[Kafka: OTP] ← Короткая очередь (TTL=5 минут, auto-drop старых)
        ↓
[Приоритетная обработка]
```

**Преимущества 2 очередей**:
- ✅ Одноразовые пароли обрабатываются в приоритете
- ✅ Старые OTP автоматически дропаются
- ✅ После восстановления новые OTP отправляются сразу
- ✅ Уведомления не блокируют срочные сообщения

## Неустойчивость IT-систем

### Опрос: IT-системы устойчивые или неустойчивые?

**Результаты голосования**:
- **Неустойчивые**: ~70%
- **Устойчивые**: ~30%

### Концепция из Нассима Талеба (Antifragile)

**Два типа систем по реакции на воздействия**:

#### Хрупкие (Неустойчивые) системы
- Любое воздействие → **ухудшение**
- Не способны адаптироваться
- Накапливают повреждения

**Примеры**: Деревянная мебель, стекло, неживые объекты

#### Антихрупкие (Устойчивые) системы
- Умеренные воздействия → **укрепление**
- Способны адаптироваться
- Становятся сильнее от стресса

**Примеры**: Мышцы (после спортзала становятся сильнее), иммунная система

### IT-системы — крайне неустойчивые

**Почему**:
```
Трафик увеличился → Система упала (не "нарастила процессоры")
Один процессор сломался → Производительность упала (не "отрастила новый")
Прошло время → Софт устарел, уязвимости появились
```

**Вывод**: 
> IT-системы не приспосабливаются сами. Любое воздействие несёт скрытую опасность. Сделать их надёжными — очень сложная задача.

## Проблемы асинхронных архитектур

### Типичная архитектура с Kafka

```
[Мобильное приложение]
        ↓
[BFF (Backend for Frontend)]
        ↓
[Kafka очередь 1]
        ↓
[Антифрод система] ← Проверка на мошенничество
        ↓
[Kafka очередь 2]
        ↓
[Скоринг система] ← Оценка кредитоспособности
        ↓
[Kafka очередь 3]
        ↓
[Движок принятия решений]
        ↓
[Kafka очередь 4]
        ↓
[Информирование пользователя (SMS)]
        ↓
[Сотовые сети]
```

**На первый взгляд**: Архитектура отказоустойчивая
- ✅ Если Скоринг упадёт, BFF продолжит принимать заявки
- ✅ Kafka хранит сообщения на диске
- ✅ После восстановления обработка продолжится

### Проблема с точки зрения надёжности

#### Что сложно понять в такой архитектуре:

**1. Потеряли ли мы заявку?**
```
Заявка попала в BFF
Записалась в Kafka 1
Антифрод прочитал? Не прочитал? Зависла?
Скоринг обработал? Не обработал?
→ Непонятно
```

**2. Какой статус показать пользователю?**
```
Пользователь: "Где моя заявка?"
Система: "Emmm... она где-то в процессе..."
```

**3. Залипла ли заявка на каком-то шаге?**
```
Заявка в Kafka 2 уже 3 часа
Это нормально? Система перегружена? Или сломалась?
```

**4. Как быстро обнаружить сбой?**
```
Скоринг сломался час назад
Симптомы:
  - Kafka 2 накапливает сообщения ← Но это видно не сразу
  - Пользователи не получают ответы ← Но они ждут и не жалуются
  - Через 2 часа всё заполнится → Поздно
```

### Почему это проблемно для надёжности?

**Нарушается пункт 2.1**: Быстро обнаружить
```
❌ Сбой виден через 1-2 часа (когда накопится много)
❌ Или когда пользователи начнут жаловаться
❌ Непонятно, на каком именно этапе проблема
```

### Reliability-Driven Architecture

**Если архитектура думает про надёжность**, нужно сразу проектировать:

**1. Как узнать, что ничего не залипло?**
```
Каждая заявка:
  - Timestamps для каждого этапа
  - Timeout detection (если заявка старше N минут на этапе → алерт)
```

**2. Как понять статус заявки?**
```
Единая таблица состояний:
  application_id | stage          | timestamp           | status
  123           | antifraud      | 2024-09-11 10:00   | processing
  123           | scoring        | 2024-09-11 10:05   | processing
  456           | decision       | 2024-09-11 10:10   | completed
```

**3. Как гарантировать, что не потеряли заявку?**
```
End-to-end мониторинг:
  - Счётчик заявок на входе
  - Счётчик заявок на каждом этапе
  - Счётчик успешно завершённых
  
  Если входящие ≠ завершённые + в_процессе → проблема
```

**4. Как быстро обнаружить зависшие заявки?**
```
SELECT COUNT(*) FROM applications
WHERE stage = 'scoring'
  AND timestamp < NOW() - INTERVAL '10 minutes'

Если > 0 → Алерт: заявки зависли на скоринге
```

## Принципы работы с надёжностью

### Принцип 1: Надежда — не стратегия

**Hope is not a strategy** (девиз Google SRE)

**Антипаттерн**:
```
"Авось пронесёт"
"Это вряд ли случится"
"Мы уже 2 года без сбоев, значит всё хорошо"
```

**Правильно**:
```
Если последствия нежелательны → готовимся заранее
Даже если вероятность кажется низкой
```

**Реальный пример** (7 лет назад):
> Лектор настоял на написании своих инструментов для алертинга вместо покупки зарубежных.
> 
> Вопрос: "Зачем?"
> 
> Ответ: "Telegram уже пытались банить. Неизвестно, что будет с зарубежным софтом."
> 
> Результат: Сейчас самописные инструменты спасают — зарубежные заблокированы.

### Принцип 2: Проблема в проде = наивысший приоритет

**Очевидно?** Оказывается, нет.

**Что лектор встречал**:
- ❌ Люди стесняются уйти с совещания
- ❌ Прерывают звонок, но не сразу
- ❌ Доканчивают собеседование с кандидатом
- ❌ **Самое страшное**: Ушёл на обед во время починки сбоя 🤦

**Правильное поведение**:
```
1. Звонок/алерт о проблеме в проде
2. Извиниться и немедленно выйти
3. Начать работу над проблемой
4. Еда, сон, встречи — ПОСЛЕ
```

**Что это предполагает**:
- Культуру компании, где это приемлемо
- Дежурные не назначают внешние встречи в дни дежурства
- Коллеги знают, что дежурный может внезапно уйти

### Принцип 3: Готовиться к устранению сбоев надо заранее

**Что делать заранее**:
- Логи (правильные, детальные)
- Мониторинг
- Автоматизация восстановления
- Обучение команды
- Репетиции (chaos engineering)
- Бэкапы
- Runbook'и (инструкции на случай сбоя)
- Продуманная архитектура

**Архитектура — это тоже подготовка**:
> Когда вы продумываете отказоустойчивую архитектуру, вы заранее готовитесь к сбоям.

**Вывод**: 
> Готовый инженер работает быстрее и лучше, чем неготовый.

### Принцип 4 (Антипаттерн): "Работает — не трогай"

**Почему это антипаттерн**:

❌ **Нарушает Принцип 1**: Надежда — не стратегия
```
"Не трогаем" = "Надеемся, что не сломается само"
Но система неустойчива → рано или поздно сломается
```

❌ **Нарушает Принцип 3**: Не готовы к устранению
```
Давно не трогали → Забыли, как это работает
Скрипты деплоя устарели
Зависимости устарели
Когда сломается → будем долго чинить
```

**Статистика**:
> ~70% сбоев связаны с изменениями. НО это не значит, что надо прекратить изменения!

**Парадокс**:
```
Больше изменений → Больше сбоев → Но быстро чиним (знаем систему)
Меньше изменений → Меньше сбоев → Но катастрофично долго чиним
```

**Решение современных практик**:
- DevOps пропагандирует частые релизы
- Continuous Deployment
- Автоматизация → снижает риск ошибок при релизах

**Пример из Google Cloud**:
> Если у сервиса не было новых коммитов 6 месяцев → робот начинает спамить разработчикам: "Пересоберите, чтобы проверить, что система всё ещё собирается и деплоится."

### Принцип 5: Простое, но рабочее > Сложное, но идеальное

**Контекст**: Споры о том, нужна ли надёжность

**Правило**:
> Работающий простой сервис лучше, чем нереализованный сложный с кучей фич.

**Но**:
- Для стартапа: фичи важнее надёжности (иначе не выиграете конкуренцию)
- Для зрелого бизнеса: надёжность критична

**Закон**:
> С увеличением сложности → Увеличивается количество сбоев

**Вывод**: Не добавляйте сложность без необходимости.

### Принцип 6: Каждый сбой — учебный материал

**Единственный способ** жить с неустойчивыми системами:

```
Сбой → Анализ → Обучение → Улучшение → Меньше сбоев
```

**Nassim Taleb (Antifragile)**:
> Если неприятность вас не убила, и вы из неё что-то извлекли → вы стали сильнее.

**Применительно к SRE**:
> Сбои нужны. Они нас учат. Без сбоев мы не знаем, где слабые места.

## Главное о людях и ошибках

### Самый главный источник сбоев

**Вопрос**: Что является главным источником сбоев?

**Ответ**: **Люди. Все ошибаются.**

### Реальные истории

#### История 1: Опытный инженер и баг в прошивке

**Что произошло**:
```
Задача: Перенастроить маршрутизатор (изменить 1 команду)
Процедура: Завести работы, написать план, делать ночью
Инженер: Опытный, "это простая команда, сделаю быстро днём"

Результат:
  1. Ввёл правильную команду
  2. На правильном маршрутизаторе
  3. Но в этой версии прошивки был баг
  4. Команда привела к полному отказу маршрутизатора
  5. Крупный сбой днём (вместо мелкого ночью)
```

**Вывод**: Даже опытные ошибаются. Процедуры созданы не просто так.

#### История 2: Стажёр и продакшен база

**Что произошло**:
```
Стажёр запустил тесты
Тесты делают: DROP ALL TABLES для чистоты
Проблема: Запустил против продакшен базы (она была доступна всем)

Результат: Удалили все данные → восстановление из бэкапов
```

**Две ошибки**:
1. Стажёр не проверил, куда направлены тесты
2. **Системная**: Продакшен база была доступна всем (нет защиты на уровне инфраструктуры)

### Термин "Человеческий фактор" — бесполезен

**Почему лектор не любит**:
```
"Сбой из-за человеческого фактора" = "Кто-то ошибся"

Вопрос: Что делать?
Ответ: ???

→ Не даёт никакой полезной информации
→ Не помогает предотвратить повторение
```

**Правильно**:
```
❌ "Человеческий фактор"
✅ "QA не обнаружил проблему в конкретном сценарии"
✅ "Тесты не запускались для ускорения процесса"
✅ "Продакшен база была доступна всем разработчикам"
✅ "Не было подтверждения перед удалением в UI"
```

**Конкретная проблема** → **Конкретное решение**

### Blameless Culture (Культура без обвинений)

**Что НЕ надо делать**:
```
❌ "Как ты мог?!"
❌ "Из-за тебя сбой!"
❌ Эмоциональное давление для "наказания"
❌ Увольнять за первую ошибку
```

**Что НАДО делать**:
```
✅ Понять, в какой момент произошла ошибка
✅ Понять, как другие могут рассуждать аналогично
✅ Изменить процессы, чтобы снизить вероятность
✅ Изменить инструменты (добавить проверки)
✅ Обучить команду
```

**Принцип**:
> Если человек адекватный, обученный, опытный, но ошибся — это не его вина. Люди ошибаются — это известный факт. Надо менять систему, а не винить человека.

**НО**: Это не значит "прощать всё всем"
- Повторяющиеся ошибки → проблема обучения
- Игнорирование процедур → проблема дисциплины
- Но эмоции и крик не помогут

### Классификация причин сбоев

**Было в компании** (7 лет назад):
```
Причины инцидентов (выбор из списка):
  - Проблема в коде
  - Проблема в инфраструктуре
  - Человеческий фактор ← Очень часто выбирали
  - ...
```

**Вопрос**: Что делать с "человеческим фактором"?

**Ответ**: Уволить всех людей? 🤷

**Сейчас**:
```
Причины (конкретные):
  - Тесты не обнаружили edge case
  - Отсутствовала проверка в UI
  - Процедура не соблюдена: пропущен шаг X
  - Недостаточно логов для диагностики
  - Не было runbook для этого сценария
```

## Выводы и инсайты

- **Надёжность = Редко + Быстро**: Работаем либо над тем, чтобы ломалось реже, либо над тем, чтобы чинилось быстрее

- **Быстро ≠ Только скорость починки**: Включает обнаружение, диагностику, восстановление И устранение последствий

- **IT-системы крайне неустойчивы**: Не адаптируются, не "отращивают процессоры", любое воздействие несёт опасность

- **Восстановление ≠ Устранение последствий**: Пример со SMS — система "работает", но пользователи ещё час не получают нужные сообщения

- **Две очереди лучше одной**: Разделение срочных и несрочных задач — фундаментальный паттерн надёжности

- **Асинхронность = проблемы с наблюдаемостью**: Kafka удобна, но сложно понять: "Где сейчас заявка? Залипла или обрабатывается?"

- **Reliability-Driven Architecture**: Архитектура, думающая про надёжность, сразу закладывает мониторинг состояния

- **"Надежда — не стратегия"**: Готовимся к тому, что кажется маловероятным, если последствия критичны

- **"Работает — не трогай" — антипаттерн**: Приводит к катастрофическим сбоям, когда что-то всё-таки сломается

- **~70% сбоев от изменений**: Но это не значит, что надо прекратить изменения — надо научиться чинить быстро

- **Все ошибаются. Всегда**: Опытные, новички, осторожные, внимательные — все. Система должна это учитывать.

- **"Человеческий фактор" — бесполезный термин**: Не даёт никакой информации для улучшения. Ищите конкретные причины.

- **Blameless culture ≠ безнаказанность**: Не винить эмоционально, но анализировать и улучшать процессы

- **Каждый сбой — учебный материал**: Единственный способ жить с неустойчивыми системами — учиться на ошибках

- **Сбои нужны**: Без них мы не знаем слабые места и не готовы к реальным критичным ситуациям

- **Простое > Сложное**: Работающий простой сервис лучше нереализованного сложного с фичами
