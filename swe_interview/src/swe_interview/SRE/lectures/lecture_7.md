## О чем лекция
Лекция посвящена различным подходам к хранению данных, типам баз данных и компромиссам между производительностью, масштабируемостью и согласованностью данных.
## Ключевые вопросы
- Какие существуют типы хранилищ и баз данных?
- Что такое ACID и BASE, и в чем их принципиальная разница?
- Как репликация и шардирование влияют на производительность?
- Какие компромиссы приходится делать при выборе хранилища?

---
## Основная часть

### Варианты хранения данных

**Базовые уровни хранения**:
- **На диске** — персистентное хранение
- **В памяти (RAM)** — быстрое, но энергозависимое хранение
- **В базе данных** — структурированное хранение с дополнительными гарантиями

---

### Типы баз данных

| Тип БД | Описание | Примеры |
|--------|----------|---------|
| **Реляционные** | Таблицы со связями, SQL | PostgreSQL, MySQL, Oracle |
| **Нереляционные (NoSQL)** | Гибкая схема, горизонтальное масштабирование | MongoDB, Cassandra |
| **Векторные** | Хранение и поиск векторных представлений | Pinecone, Milvus, Weaviate |
| **Key-Value** | Простейшая модель: ключ → значение | Redis, Memcached, DynamoDB |
| **Файловые** | Хранение файлов | S3, HDFS, MinIO |

---

### Характеристики баз данных

#### Схема данных

**Со схемой (Schema-based)**:
- Жесткая структура таблиц и типов данных
- Миграции при изменении структуры
- Примеры: PostgreSQL, MySQL

**Без схемы (Schema-less)**:
- Гибкая структура документов
- Легко изменять структуру
- Примеры: MongoDB, Elasticsearch

#### Язык запросов

- **SQL** — декларативный язык для реляционных БД
- **NoSQL специфичные** — MongoDB Query Language, Cypher (Neo4j), и т.д.
- **API-based** — REST или gRPC интерфейсы

---

### Локальность данных (Data Locality)

**Row-oriented (построчное хранение)**:
```
[ID | Name | Age | City]
[1  | Alice| 25  | NYC ]
[2  | Bob  | 30  | LA  ]
```
- Оптимально для OLTP (частые вставки/обновления)
- Примеры: PostgreSQL, MySQL

**Column-oriented (постолбцовое хранение)**:
```
ID:   [1, 2, 3, ...]
Name: [Alice, Bob, Charlie, ...]
Age:  [25, 30, 35, ...]
```
- Оптимально для OLAP (аналитические запросы)
- Лучшее сжатие однородных данных
- Примеры: ClickHouse, Apache Parquet

| **Персистентные БД** | Есть (на диск) | PostgreSQL, MongoDB |
| **In-Memory БД** | Нет (или опционально) | Redis, Memcached |

### Репликация

**Назначение**: Хранение нескольких копий данных для отказоустойчивости и масштабирования чтения.

**Влияние на производительность**:
- ✅ Ускоряет чтение (можно распределить нагрузку)
- ⚠️ Замедляет запись (нужно синхронизировать реплики)

**Поддержка**:
- Встроенная (из коробки): PostgreSQL, MySQL, MongoDB
- Требует дополнительной настройки: некоторые NoSQL решения

**Схема репликации**:
```
        [Master]
         /  |  \
        /   |   \
   [Replica] [Replica] [Replica]
   
   Запись → Master
   Чтение → Любая реплика
```

---

### Шардирование (Партиционирование)

**Принцип**: Разделение данных на части и хранение на разных серверах.

**Варианты разбиения**:
- По **диапазону значений**: пользователи с ID 1-1000 на сервер 1, 1001-2000 на сервер 2
- По **хешу**: `hash(user_id) % N_servers`
- По **географии**: данные европейских пользователей в EU, американских в US

**Примеры**:
```
Вертикальное (по столбцам):
Сервер 1: [user_id, name, email]
Сервер 2: [user_id, profile_data, settings]

Горизонтальное (по строкам):
Сервер 1: пользователи 1-10000
Сервер 2: пользователи 10001-20000
```

**Нюансы**:
- В общем случае сложно реализовать универсально
- Требует индивидуального подхода под конкретную задачу
- JOIN-ы между шардами становятся очень дорогими

---

### Оптимизация под запись или чтение

| Характеристика | Оптимизация под запись | Оптимизация под чтение |
|----------------|------------------------|------------------------|
| **Write latency** | Низкая | Высокая |
| **Read latency** | Высокая | Низкая |
| **Примеры** | Cassandra, ScyllaDB | Elasticsearch, ClickHouse |
| **Use case** | Логи, метрики, IoT | Аналитика, отчеты |

**Холодные хранилища**:
- Данные хранятся "на всякий случай"
- Очень медленное чтение
- Низкая стоимость хранения
- Примеры: AWS Glacier, Google Coldline

### ACID vs BASE

#### ACID — гарантии строгой согласованности

**A — Atomicity (Атомарность)**
- Транзакция либо выполняется полностью, либо не выполняется вообще
- Нет промежуточных состояний
- Пример: перевод денег (списание и зачисление — одна операция)

**C — Consistency (Консистентность)**
- БД всегда находится в валидном состоянии
- Соблюдаются все ограничения (constraints):
  - Внешние ключи (foreign keys)
  - Уникальность (unique)
  - Проверки значений (check constraints)
- Ни одна операция не может нарушить правила

**I — Isolation (Изоляция)**
- Конкурентные транзакции изолированы друг от друга
- Не видим частичного применения других транзакций
- Пример: при долгом чтении не видим изменений, сделанных параллельной транзакцией

**D — Durability (Долговечность)**
- Подтвержденные транзакции сохраняются навсегда
- Даже перезапуск системы не приведет к потере данных
- Данные гарантированно записаны на диск

**Примеры ACID БД**: PostgreSQL, MySQL, Oracle

---

#### BASE — гибкость вместо строгой согласованности

**BA — Basically Available (В основном доступна)**
- Система остается доступной даже при частичных сбоях
- Приоритет на доступность, а не на консистентность

**S — Soft State (Мягкое состояние)**
- Состояние может быть нечетким
- Эффект изменений видим не сразу
- Можно читать "устаревшие" данные

**E — Eventual Consistency (Финальная согласованность)**
- Система со временем стабилизируется
- Гарантируется согласованность в будущем, но не немедленно
- Аналогия: в банках итоги дня подбиваются к вечеру

**Примеры BASE БД**: Cassandra, DynamoDB, Riak

---

### NoSQL и NewSQL

**NoSQL**
- Все БД, которые жертвуют какими-то гарантиями ACID
- Приоритет: масштабируемость и производительность
- Часто следуют принципам BASE

**NewSQL**
- Попытка объединить SQL и NoSQL
- Цель: ACID-гарантии + горизонтальное масштабирование
- Примеры: CockroachDB, Google Spanner, VoltDB

**Компромисс**:
- ACID-гарантии замедляют работу
- NoSQL работает быстрее, но с меньшими гарантиями
- Золотой пули пока не существует — выбор зависит от задачи

## Выводы и инсайты

- **Нет универсального решения**: Выбор хранилища зависит от конкретных требований — согласованность vs производительность, чтение vs запись, структурированность vs гибкость

- **CAP-теорема в действии**: Невозможно одновременно гарантировать Consistency, Availability и Partition tolerance — приходится выбирать

- **ACID для критичных данных**: Платежи, заказы, финансы требуют строгих гарантий — используйте ACID БД

- **BASE для высоконагруженных систем**: Логи, метрики, аналитика могут пожертвовать немедленной консистентностью ради скорости

- **Репликация ≠ Шардирование**: 
  - Репликация — для отказоустойчивости и масштабирования чтения
  - Шардирование — для масштабирования данных и записи

- **Локальность данных имеет значение**: Row-oriented для OLTP, Column-oriented для OLAP

- **NewSQL — это компромисс**: Пытается решить проблему масштабирования ACID БД, но с ценой дополнительной сложности

- **Начните с простого**: Используйте PostgreSQL, пока не появится явная необходимость в чем-то другом
