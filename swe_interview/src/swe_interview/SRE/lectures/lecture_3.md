## О чем лекция
Лекция посвящена фундаментальным проблемам распределённых систем, проблеме консенсуса, CAP и PACELC теоремам, а также практическим способам размена консистентности на доступность и скорость.

## Ключевые вопросы
- Что считать распределённой системой?
- Что такое проблема консенсуса и зачем её решать?
- Что утверждает CAP теорема и как её понимать?
- Когда предпочитают доступность консистентности?
- Как кэширование меняет свойства системы?

---

## Основная часть

### Определение распределённой системы

**Практическое определение для курса**:

> Распределённая система — это всё, что **за корпусом компьютера**.

**Почему такое широкое определение**:

```
Внутри корпуса компьютера:
  - CPU, RAM, диски в одной машине
  - Связь надёжная
  - Сбои редки
  
За корпусом (сеть):
  - Между серверами в дата-центре
  - Между мобильным приложением и сервером
  - Между микросервисами
  - Связь ненадёжная
  - Сбои часты
```

**Что включаем в распределённую систему**:

```
[Мобильные приложения] ← Часть системы!
        ↕ (сеть)
[Веб-приложения в браузере] ← Часть системы!
        ↕ (сеть)
[Серверы в дата-центре]
  ├─ Микросервис 1
  ├─ Микросервис 2
  ├─ База данных (несколько нод)
  └─ Очереди сообщений
        ↕ (сеть)
[Другие дата-центры]
```

**Почему мобильное приложение — часть системы**:
- Пользователь может уехать в лес (нет связи)
- Спуститься в метро (плохая связь)
- Переключаться между WiFi и мобильной сетью
- Ноутбуки подключаются/отключаются
- Выезд за границу (роуминг, другие сети)

**Вывод**: Сеть между клиентом и сервером **крайне ненадёжна**.

## Проблема консенсуса

### Что это такое?

**Определение**: 
> Проблема консенсуса — когда распределённой системе надо **договориться о каком-то одном значении**.

**Примеры в жизни**:
- Родителям надо договориться об имени ребёнка
- 10 людям в команде надо выбрать общую цель
- Оркестру надо синхронизироваться, когда начать играть

**Примеры в IT**:
- Договориться об идентификаторе объекта
- Выбрать главного (master election)
- Согласовать порядок событий
- Решить, кто обрабатывает задачу

### Почему договориться сложно?

**Проблемы в команде из 10 человек**:
- Кто-то пропустил встречу (болезнь, отпуск)
- Кто-то был в телефоне
- Кто-то в командировке
- Записали в Jira, но не все прочитали
- → Всегда кто-то чего-то не знает

**Проблемы в IT-системах**:
- Связь может пропасть
- Сообщения могут потеряться
- Ноды могут упасть
- Задержки в сети
- → Гарантировать синхронизацию очень сложно

### Способы решения консенсуса

#### 1. Избегать консенсуса (лучший способ)

**Пример**: UUID/ULID для идентификаторов

```
Проблема:
  Два сервера хотят создать объект
  Нужен уникальный ID
  Как не конфликтовать?

Плохое решение (требует консенсуса):
  ID = счётчик (1, 2, 3, ...)
  Нужно договариваться, кто какой номер берёт

Хорошее решение (без консенсуса):
  ID = UUID (каждый генерирует локально)
  Коллизии практически невозможны
  Не нужно договариваться
```

**Алгоритмы для уникальных ID без согласования**:
- UUID (128-bit случайное значение)
- ULID (время + случайность)
- Snowflake ID (timestamp + worker_id + sequence)

#### 2. Выбрать одного главного (master)

**Жизненная аналогия**: Дирижёр в оркестре

```
Без дирижёра:
  Музыканты должны постоянно синхронизироваться
  Сложно начать играть одновременно
  
С дирижёром:
  Все смотрят на одного
  Он даёт сигнал → все начинают
  Проще и быстрее
```

**В IT**:
```
[Master Server] ← Все спрашивают его
  - Выдаёт ID
  - Решает конфликты
  - Источник истины
```

**Проблема**: Master может упасть → нужен новый master

**Решение**: Периодически выбирать нового master (master election)

#### 3. Использовать distributed consensus алгоритмы

**Алгоритмы** (готовые, не нужно писать самому):
- Paxos
- Raft
- Zab (используется в ZooKeeper)

**Системы, решающие консенсус**:
- ZooKeeper
- etcd
- Consul

**Когда использовать**: Когда нужна отказоустойчивость и нельзя иметь single master.

### Примеры видимого консенсуса

#### Git (система контроля версий)

**Проблема**: Много разработчиков изменяют код

**Решение Git**:
- Нет центрального мастера (как в Jira)
- Каждый работает локально
- Синхронизация по необходимости
- Конфликты разрешаются при merge

**Свойство**: Eventual consistency

#### Blockchain

**Проблема**: Договориться о транзакциях без центрального арбитра

**Решение**: Алгоритмы консенсуса (Proof of Work, Proof of Stake)

**Применение**: Финансовые транзакции без доверенной стороны

## CAP теорема

### Три свойства

#### C — Consistency (Консистентность)

**Что значит**:

> Если клиент сделал **запись** в систему и получил подтверждение, то **любое последующее чтение** из любого узла системы вернёт эту запись (учтёт все её эффекты).

**Пример — консистентная система**:
```
1. Клиент записал в БД ноду 1: "balance = 100"
2. Получил подтверждение: "OK"
3. Нода 1 упала
4. Клиент читает из ноды 2
5. Видит: "balance = 100" ✅

→ Консистентно
```

**Пример — НЕ консистентная система**:
```
1. Пользователь отправил деньги другу
2. Получил: "Отправлено"
3. Смотрит в ленту операций: транзакция есть ✅
4. Смотрит на баланс: сумма не изменилась ❌

→ НЕ консистентно (разные части системы рассинхронизированы)
```

#### A — Availability (Доступность)

**Что значит**:

> Система **отвечает на запрос** без ошибки (не возвращает "service unavailable").

**Важно**: Если система вернула "некорректный запрос" — это тоже доступность (она правильно ответила).

**НЕ доступность**: Если система вернула ошибку типа "база недоступна", "нет соединения".

#### P — Partition Tolerance (Устойчивость к сетевым проблемам)

**Что значит**:

> Система продолжает работать, даже если связь между некоторыми узлами прервалась.

**Пример**:
```
[Нода 1] ─╳─ [Нода 2]  ← Связь прервалась

Partition Tolerance = система работает, несмотря на разрыв
```

### Утверждение CAP

**CAP теорема**:

> При наличии сетевых проблем (Partition) можно гарантировать **только одно**:
> - Либо **C** (Consistency)
> - Либо **A** (Availability)

**Невозможно**: C + A + P одновременно

**Когда сети хорошо**: Можно C + A одновременно

### Визуализация

```
Нет проблем с сетью:
  ✅ Consistency + Availability → Возможно

Проблемы с сетью:
  ✅ Consistency + Partition Tolerance (CP)
     ИЛИ
  ✅ Availability + Partition Tolerance (AP)
  
  ❌ C + A + P → Невозможно
```

## PACELC теорема (расширение CAP)

### Формулировка

**PACELC** читается как:

> **P**artition:
>   - Либо **A** (Availability)
>   - Либо **C** (Consistency)
>
> **E**lse (когда сети хорошо):
>   - Либо **L** (Latency - скорость)
>   - Либо **C** (Consistency)

### Что добавляет PACELC?

**Insight**: Даже когда сети хорошо, **консистентность замедляет работу**.

**Почему**:
```
Консистентная система:
  1. Запись идёт в несколько узлов
  2. Ждём подтверждения от всех
  3. Только потом отвечаем клиенту
  → Медленно

Быстрая система:
  1. Записали в один узел
  2. Сразу ответили клиенту
  3. Синхронизация с другими узлами — потом
  → Быстро, но не консистентно (пока)
```

**Пример в жизни**:
```
10 друзей работают консистентно:
  - Постоянно на созвоне
  - Каждый шаг согласовывают
  - Все всегда в курсе
  → Медленно

10 друзей работают быстро:
  - Работают независимо большую часть времени
  - Синхронизируются периодически
  - Есть моменты рассинхронизации
  → Быстро
```

## Практические примеры trade-offs

### Мобильные приложения: сеть ненадёжна

**Сценарий**: Мобильное приложение для заметок

**Вариант 1: Консистентность (CP)**:
```
1. Пользователь создаёт заметку
2. Приложение отправляет на сервер
3. Нет связи → "Ошибка. Попробуйте позже"
4. Пользователь не может работать

→ Консистентно, но недоступно без связи
```

**Вариант 2: Доступность (AP)**:
```
1. Пользователь создаёт заметку
2. Сохраняется локально (кэш)
3. "Заметка создана" (даже без связи)
4. При появлении связи → синхронизация

→ Доступно всегда, но временно не консистентно
```

**Что предпочитают пользователи**: Вариант 2 (AP)

**Почему**: Лучше работать офлайн с локальными данными, чем вообще не работать.

### Банковские приложения: баланс vs переводы

**Реальный опыт лектора**:
```
Московское метро:
  1. Приложил карту
  2. Прошёл через турникет (доступность ✅)
  3. Уведомление: "Списано 60₽"
  4. Смотрю в приложение: транзакции нет ❌
  5. Через несколько часов: транзакция появилась ✅
  
→ Доступность > Консистентность
```

**Почему это приемлемо**:
- Проход в метро критичен (сейчас)
- Отображение транзакции в истории — некритично (потом увижу)
- Пользователя это не волнует

**Другой пример**:
```
Перевод другу:
  1. Нажал "Отправить 1000₽"
  2. "Отправлено" ✅
  3. Смотрю в ленту: транзакция есть ✅
  4. Смотрю на баланс: не изменился ❌
  5. Через минуту: баланс обновился ✅
  
→ Временная несогласованность, но работает
```

**Вывод лектора**:
> Даже в банковской сфере люди часто предпочитают доступность консистентности — хотят, чтобы работало сейчас.

## Кэширование: главный способ размена

### Базовая архитектура без кэша

```
[Клиент] → [Сервер] → [SQL База данных]
                         ↓
                    Консистентно
                    Но медленно при нагрузке
```

### С кэшем (типичное решение)

```
[Клиент] → [Сервер] → [Memcached/Redis] ← Быстро, но устаревшие данные
                  ↘
                   [SQL База данных] ← Медленно, но точные данные
```

**Эффект**:
```
Запись:
  1. Клиент пишет в БД
  2. БД подтверждает
  3. Инвалидация кэша (опционально)
  
Чтение (при падении БД):
  → Читаем из кэша
  → Система доступна ✅
  → Но данные могут быть устаревшими ❌
```

**Trade-off**:
- ✅ Доступность повышается (работает даже если БД упала)
- ✅ Скорость возрастает (кэш быстрее БД)
- ❌ Консистентность снижается (кэш может быть устаревшим)

### Варианты кэширования

#### Вариант 1: Кэш всегда, консистентность потом

```
Чтение:
  → Всегда из кэша
  → Быстро ✅
  → Могут быть устаревшие данные

Запись:
  → В БД
  → Инвалидация кэша в фоне
  → Асинхронно
```

**Когда**: Соцсети, новостные ленты, некритичные данные

#### Вариант 2: Кэш с валидацией (консистентный кэш)

```
Чтение:
  1. Проверяем кэш
  2. Валидируем с мастером (свежесть данных)
  3. Если устарел → читаем из БД, обновляем кэш
  
Запись:
  4. Пишем в БД (master)
  5. Инвалидируем ВСЕ кэши
  6. Только после этого → подтверждение клиенту
  → Медленно, но консистентно
```

**Когда**: Критичные данные (баланс счёта, inventory)

## Исторические примеры: пластиковые карты

### Протяжка карты через терминал

**Как работало** (давно):
```
1. Карточка с выпуклыми цифрами и именем
2. Клали на планшет (не iPad, а пластина)
3. Накрывали копировальной бумагой
4. Протягивали специальную ручку
5. Цифры отпечатывались на бумажке
6. Вручную писали сумму
7. Клиент ставил подпись
8. Вечером → бухгалтер относил бумажки в банк
9. Банк обрабатывал транзакции
```

**Свойства**:
- ✅ Availability: Очень высокая (работает везде, без связи)
- ❌ Consistency: Очень низкая (банк узнаёт о транзакции через день)
- ✅ Latency: Моментальная оплата для клиента

**Почему это был прогресс**:
> Не нужно носить наличные. Банк гарантирует оплату по предъявлении карты.

### Зачем выпуклые цифры на карте?

**Вопрос**: Почему цифры именно выпуклые?

**Ответ**: Чтобы делать оттиск на бумаге (механический способ копирования).

**Почему держались долго**: Совместимость — старые терминалы (протяжка) продолжали работать.

**Сейчас**: Цифры всё ещё есть на некоторых картах, но не выпуклые.

### Терминалы в самолётах

**Сценарий**: Оплата картой на борту самолёта (до повсеместного WiFi)

**Как работало**:
```
1. Терминал сохранял информацию о транзакции локально
2. Самолёт приземлился
3. Терминал подключился к сети
4. Передал все транзакции в банк
5. Банк обработал
```

**Trade-off**: Доступность в полёте, консистентность потом.

### Проверка карты по телефону

**Дополнительная защита** (когда сумма подозрительная):
```
1. Продавец принял карту
2. Позвонил в банк по номеру на карте
3. Назвал номер карты
4. Банк проверил: "Есть ли деньги? Не украденная ли карта?"
5. Банк разрешил или запретил транзакцию
```

**Эффект**: 
- Консистентность повысилась (проверка актуального баланса)
- Скорость транзакции упала (ждём звонка)

## Что предпочитают пользователи?

### Общий паттерн: Availability > Consistency

**Банковские приложения**:
```
"Баланс посмотрю потом, перевод сделай сейчас"
```

**Мессенджеры**:
```
Показать закэшированные сообщения лучше, чем:
  - Не показывать ничего
  - Долго грузить с сервера
```

**Интернет-магазины**:
```
Хочется:
  - Если товар в каталоге → точно есть на складе (consistency)
  
Реальность:
  - Товар в каталоге, добавил в корзину
  - При оплате: "Извините, товара нет" ❌
  - → Не консистентно, но доступность высокая
```

**Почему так делают**: Проверять склад при каждом просмотре = медленно и дорого.

### Latency vs Consistency

**Даже когда сети хорошо**, пользователи выбирают скорость:

**Соцсети (VK, Instagram)**:
```
Хотелось бы:
  - Видеть ВСЕ свежайшие посты друзей (консистентно)
  
Реальность:
  - Показываем быстро
  - Посты могут появиться с задержкой
  - Eventual consistency
  
→ Скорость > Консистентность
```

## Архитектурные следствия

### Инженеры думают не так

**Наблюдение лектора**:
> Инженеры часто начинают с проектирования консистентных систем, а потом переписывают для доступности и скорости.

**Почему так**:
- Влияние SQL баз данных (ACID гарантии)
- Много книг про транзакции
- Учат про важность консистентности

**История**: Популярность MySQL, PostgreSQL → мышление "всё должно быть в транзакции"

**Рекомендация**:
> **По умолчанию** проектируйте систему доступной и быстрой.
> **Консистентность** добавляйте только там, где действительно критично.

### Reliability-Driven Architecture

**Если архитектура думает про надёжность**, то:

**Вопросы при проектировании**:
- Как это будет работать **при проблемах с сетью**?
- Что важнее: консистентность или доступность?
- Нужна ли консистентность **везде** или только в критичных местах?
- Как быстро обнаружим проблему?
- Можно ли работать без distributed consensus?

## Практический совет по применению

### Где нужна консистентность

**Критичные места**:
- Финансовые транзакции (списание/зачисление)
- Inventory (остатки товара при оплате)
- Бронирование (билеты, отели)
- Аукционы

### Где можно пожертвовать

**Некритичные места**:
- Лента операций (eventual consistency)
- Новостная лента
- Комментарии и лайки
- История сообщений
- Статистика и аналитика

### Осознанный выбор

**При проектировании каждой фичи спрашивайте**:
```
1. Что важнее: доступность или консистентность?
2. Что важнее: скорость или консистентность?
3. Что будет, если данные временно рассинхронизированы?
4. Приемлемо ли это для пользователя?
```

## Выводы и инсайты

- **Распределённая система = всё за корпусом компьютера**: Мобильные приложения, микросервисы, реплики БД — всё это распределённые системы

- **Сеть крайне ненадёжна**: Особенно между клиентом и сервером (мобильная связь, WiFi)

- **Консенсус сложен**: Избегайте, где возможно (UUID вместо счётчиков)

- **CAP: при проблемах сети выбирайте**: Либо консистентность, либо доступность

- **PACELC: даже без проблем с сетью**: Консистентность замедляет работу

- **Пользователи выбирают A и L**: Доступность и скорость важнее консистентности в большинстве случаев

- **Кэш — главный инструмент**: Самый простой способ повысить доступность и скорость (ценой консистентности)

- **Проектируйте AP по умолчанию**: Консистентность добавляйте только где критично

- **Исторический пример работает**: Пластиковые карты — 50 лет успеха с eventual consistency

- **Даже банки жертвуют консистентностью**: Метро, переводы, баланс — временная рассинхронизация приемлема

- **Инженеры ошибаются в сторону C**: Под влиянием SQL-баз проектируют слишком консистентные системы

- **Eventual consistency — норма**: В современных системах большинство данных синхронизируются не мгновенно

- **Master election — частное решение консенсуса**: Выбираем одного главного, все обращаются к нему

- **Готовые алгоритмы существуют**: Paxos, Raft, ZooKeeper — не пишите своё

- **Консистентность там, где критично**: Платежи, остатки товара, бронирования — остальное может быть eventual
