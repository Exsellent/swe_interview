## О чем лекция
Лекция посвящена механизмам репликации данных, различиям между бэкапами и репликами, выбору между синхронной и асинхронной репликацией, а также стратегиям работы с мастер-нодами и разрешению конфликтов.

## Ключевые вопросы
- В чем разница между бэкапами и репликами?
- Как работает синхронная и асинхронная репликация, и когда что использовать?
- Что делать при падении мастер-ноды?
- Как разрешать конфликты при многомастерной репликации?
- Какие подводные камни репликации и как их избежать?

## Основная часть

### Бэкапы vs Реплики

| Характеристика | Бэкапы | Реплики |
|----------------|--------|---------|
| **Изменяемость** | Неизменяемые, фиксированные | Изменяемые, "живые" копии данных |
| **Назначение** | Восстановление после катастрофы | Отказоустойчивость и масштабирование |
| **Риски** | Безопасны от случайного удаления | Можно случайно удалить важные данные |
| **Актуальность** | Snapshot на момент создания | Постоянно синхронизируются с мастером |

**Важно**: Реплика может сломаться или содержать некорректные данные, если в мастере произошла ошибка. Бэкапы защищают от таких сценариев.

---

### Цели репликации

#### 1. Отказоустойчивость на чтение

**Основной use case**: Распределение нагрузки на чтение между несколькими нодами.

```
       [Master] ← Запись
         /  |  \
        /   |   \
   [Читаем] [Читаем] [Читаем]
```

**Частный случай — иммутабельные данные**:
- Индекс интернета (обновляется раз в неделю)
- Архивные данные
- Статические ресурсы

**CDN (Content Delivery Network)**:
- Специальный случай репликации
- Кеширование статических данных по всему миру
- Географическое распределение нагрузки
- Примеры: Cloudflare, Akamai, AWS CloudFront

#### 2. Уменьшение задержки (Latency)

**Географическая близость к пользователю**:
- Размещение данных ближе к конечным пользователям
- Снижение задержки на сетевые запросы
- Экономия на трафике (локальный трафик дешевле межрегионального)

**Пример**:
```
EU Users → [EU Replica]
US Users → [US Replica]    ← Синхронизация → [Master Asia]
APAC Users → [APAC Replica]
```

### Репликация и распространение изменений

#### Архитектура: Один мастер, множество реплик

**Принцип**:
- Выбираем **одну ноду** для записи (мастер)
- Мастер распространяет изменения на реплики
- Чтение возможно из **всех нод** одновременно

```
Client (Write) → [Master] → Репликация → [Replica 1]
                                      → [Replica 2]
                                      → [Replica 3]

Client (Read) → [Любая нода]
```

### Синхронная vs Асинхронная репликация

#### Асинхронная репликация

**Как работает**:
1. Клиент отправляет запись в мастер
2. Мастер **сразу отвечает "OK"**
3. Изменения распространяются на реплики **в фоновом режиме**

**Преимущества**:
- ✅ Быстрая запись
- ✅ Устойчивость к падению реплик (если реплика упала, запись все равно проходит)
- ✅ Низкая latency для клиента

**Недостатки**:
- ⚠️ Данные распределяются **не сразу**
- ⚠️ **Риск потери данных**: мастер записал, упал, а реплики не успели получить данные
- ⚠️ Eventual consistency — пользователи могут читать устаревшие данные

**Проблемный сценарий**:
```
1. Клиент записал данные в мастер
2. Мастер подтвердил запись
3. Мастер упал до репликации
4. Запись потеряна ❌
```

**Решение**: Процедуры сверки и восстановления данных.

#### Синхронная репликация

**Как работает**:
1. Клиент отправляет запись в мастер
2. Мастер **ждет подтверждения** от всех (или кворума) реплик
3. Только после этого отвечает "OK" клиенту

**Преимущества**:
- ✅ Гарантия сохранности данных
- ✅ Все реплики актуальны
- ✅ Strong consistency

**Недостатки**:
- ⚠️ **Медленная запись** (ждем все реплики)
- ⚠️ **Если одна реплика упала** → вся запись блокируется
- ⚠️ Высокая latency для клиента

**Когда использовать**:
- Критичные данные (платежи, транзакции)
- Когда важнее надежность, чем скорость

### Добавление новой реплики

**Проблема**: Как добавить новую реплику к работающей системе?

**Процесс**:
1. Создать **snapshot** текущих данных мастера
2. Скопировать snapshot на новую ноду
3. Начать применять **лог изменений** с момента создания snapshot
4. Когда реплика "догонит" мастер — переключить в режим обычной репликации

**Сложности**:
- Нужно делать **без даунтайма** (downtime)
- Snapshot может быть очень большим
- Во время копирования данные продолжают изменяться

**Готовые решения**: Большинство современных БД имеют встроенные механизмы (например, PostgreSQL: `pg_basebackup`).

---

### Условная репликация

**Вариант 1: Репликация по условию (фильтрация)**
- Реплицируем только данные, удовлетворяющие условию
- Пример: реплицируем только активных пользователей

**Вариант 2: Потоковая репликация (streaming replication)**
- Реплицируем изменения в реальном времени по мере их появления
- Используется в большинстве современных СУБД

**Вариант 3: Кеширование (caching)**
- Частный случай репликации
- CDN — кеширование статических данных

### Что делать при падении мастера?

#### Стратегия 1: Просто поднять мастер

**Плюсы**: Сохраняем существующую конфигурацию  
**Минусы**: Может быть долго, система недоступна для записи

#### Стратегия 2: Выбрать новый мастер (failover)

**Процесс**:
1. Обнаружить падение мастера
2. Выбрать одну из реплик как новый мастер
3. Переключить клиентов на нового мастера
4. Перенастроить старые реплики на нового мастера

**Через прокси**:
```
Клиенты → [Прокси] → [Текущий мастер]
                    → [Реплика 1]
                    → [Реплика 2]
```

Прокси автоматически определяет, кто сейчас мастер, и направляет запросы.

**Требования к прокси**:
- Сама прокси должна быть отказоустойчивой (несколько инстансов)
- Прокси должны быть синхронизированы (консенсус)
- Используют системы как: HAProxy, ProxySQL, или встроенные механизмы (Patroni для PostgreSQL)

#### Проблемы при выборе нового мастера

**Проблема 1: Потеря данных**
```
Сценарий:
1. Пишем в мастер
2. Мастер падает ДО репликации
3. Запись попала только в мастер
4. Выбираем новый мастер из реплики
5. Старая запись потеряна ❌
```

**Проблема 2: Split brain (два мастера)**
```
1. Мастер временно потерял связь
2. Система решила, что он упал
3. Выбрали новый мастер
4. Старый мастер "ожил"
5. Теперь ДВА мастера! → Конфликты данных 💥
```

**Решения**:
- Использовать **кворум** при выборе мастера
- Использовать **fencing** (блокировка старого мастера)
- Тщательное тестирование сценариев failover

**Важный инсайт**: 
> Самое страшное, что все эти сценарии до конца не продуманы почти ни в одной системе. Нужно пару лет практики, чтобы понять, как работать при всех типах сбоев с БД. Поэтому обновление БД очень не любят.

### Способы репликации

#### Мейнстрим: Write-Ahead Log (WAL) репликация

**Принцип**:
1. Мастер записывает **лог всех операций** (WAL)
2. Отправляет лог на реплики
3. Реплики **применяют операции** из лога у себя
4. Мастер **не удаляет лог** до подтверждения от реплик

```
[Master] → [WAL] → Отправка → [Replica WAL] → Применение
```

**Примеры**: PostgreSQL, MySQL (binlog), MongoDB (oplog)

#### Кастомные способы

**Кеширование**:
- Записываем в кеш популярные данные
- Пример: Redis как кеш перед PostgreSQL

**Подписка на изменения (Change Data Capture)**:
- Подписка на события изменения данных
- Потоковая обработка изменений
- Примеры: Kafka Connect, Debezium

### Eventual Consistency и монотонное чтение

#### Проблема: Асинхронные реплики запаздывают

```
Время T1: Пользователь записал данные
Время T2: Пользователь читает из Реплики 1 → видит новые данные ✓
Время T3: Пользователь читает из Реплики 2 → НЕ видит данные ✗
```

**Проблема**: Пользователь "развидел" то, что уже видел раньше → плохой UX.

#### Решение: Монотонное чтение (Monotonic Reads)

**Гарантия**: То, что пользователь увидел, он **уже не "развидит"**.

**Реализация**:
- Привязать пользователя к **одной реплике** (session affinity)
- Использовать **read-your-writes** гарантии
- Читать всегда из мастера для критичных данных

**Пример с session affinity**:
```
User123 → всегда читает из Реплики 2
User456 → всегда читает из Реплики 1
```

### Многомастерная репликация (Multi-Master)

#### Зачем нужна?

**Цель**: Увеличить **скорость записи** и доступность для записи.

**Сценарии**:
- Географически распределенные системы (каждый регион имеет свой мастер)
- Офлайн-работа (каждый клиент может записывать локально, потом синхронизируется)

```
[Master EU] ↔ Синхронизация ↔ [Master US]
     ↓                              ↓
[Реплики EU]                  [Реплики US]
```

#### Проблема: Конфликты записи

**Сценарий конфликта**:
```
T1: Master EU: UPDATE users SET age=25 WHERE id=1
T1: Master US: UPDATE users SET age=30 WHERE id=1

Какое значение правильное? 🤔
```

---

### Стратегии разрешения конфликтов

#### 1. Избегание конфликтов (Conflict Avoidance)

**Подход**: Сделать так, чтобы конфликты **не возникали**.

**Как**:
- Распределить данные по мастерам (шардирование)
- Пример: Европейские пользователи редактируются только в EU мастере

```
User ID 1-10000   → Master EU
User ID 10001-20000 → Master US
```

#### 2. Last Write Wins (LWW)

**Принцип**: Побеждает последняя запись по timestamp.

```
T1 @ 10:00: age = 25
T2 @ 10:01: age = 30
Результат: age = 30 ✓
```

**Проблема**: Зависит от синхронизации часов (NTP), можно потерять данные.

#### 3. Приоритет мастеров

**Принцип**: У каждого мастера свой приоритет.

```
Master EU: priority = 1 (высший)
Master US: priority = 2

При конфликте → побеждает Master EU
```

#### 4. Кастомный алгоритм резолвинга/мержа

**Примеры**:
- Для счетчиков: суммировать инкременты
- Для списков: объединять элементы
- Бизнес-логика: применять доменные правила

```javascript
function resolveConflict(value1, value2) {
  if (typeof value1 === 'number') {
    return value1 + value2; // Суммируем
  }
  // Другая логика...
}
```

#### 5. Версионирование (как в Git)

**Принцип**: Сохранить **обе версии** в БД.

```
users:
  id: 1,
  age: [
    {version: "master-eu", value: 25},
    {version: "master-us", value: 30}
  ]
```

**Обработка при чтении**:
1. Клиент получает **обе версии**
2. Клиент (или сервис) сам **решает**, какую использовать
3. Можно показать пользователю выбор (как merge conflict в Git)

**Примеры БД с версионированием**: CouchDB, Riak

## Выводы и инсайты
- **Реплики ≠ Бэкапы**: Реплики могут сломаться вместе с мастером, бэкапы защищают от таких сценариев
- **Асинхронная vs Синхронная** — вечный компромисс:
  - Асинхронная: быстро, но eventual consistency
  - Синхронная: медленно, но гарантии сохранности
- **Failover — это сложно**: Переключение мастера требует тщательной проработки и тестирования. Split brain — реальная угроза.
- **CDN = Репликация**: Content Delivery Networks — это специализированный случай географической репликации
- **Монотонное чтение критично для UX**: Пользователь не должен "развидеть" то, что уже видел
- **Многомастерная репликация = Конфликты**: Без стратегии разрешения конфликтов multi-master развалится
- **Нет серебряной пули для конфликтов**: Выбор стратегии зависит от бизнес-логики и типа данных
- **Репликация требует опыта**: Нужны годы практики, чтобы правильно обрабатывать все edge cases. Поэтому миграции БД редки и болезненны.
- **Начните с simple master-replica**: Для большинства задач достаточно одного мастера и асинхронных реплик. Усложняйте только при реальной необходимости.
- **Тестируйте сценарии отказа**: Регулярно проводите chaos engineering и failover drills
