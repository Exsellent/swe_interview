## О чем лекция
Лекция посвящена управлению распределенными транзакциями в микросервисной архитектуре и практическому применению паттерна Saga в двух вариантах: хореография и оркестрация.

## Ключевые вопросы
- Как обеспечить согласованность данных в распределенных системах?
- Какие подходы существуют для координации длинных транзакций между сервисами?
- В чем разница между хореографией и оркестрацией?
- Как правильно реализовать оркестратор и избежать потери данных?

## Основная часть
### Двухфазный коммит (2PC)
**Принцип работы**: Сначала договариваемся, потом выполняем действие.
**Аналогия**: 
- День 1: Договариваетесь на рынке отложить трусы
- День 2: Приходите и покупаете отложенный товар
Фазы:
1. **Prepare** — все участники подтверждают готовность выполнить операцию
2. **Commit** — все участники выполняют операцию

---

### Паттерн Saga: Хореография

**Принцип**: Распределенная координация — каждый сервис знает, что делать дальше.

```
Сервис 1 → Сервис 2 → Сервис 3 → Сервис 4
                    ↘ Сервис 5
```

**Ключевые требования**:
- Для каждой операции должна быть **компенсирующая операция** (операция отмены)
- При откате операции сервис сообщает предыдущему в цепочке об ошибке
- Откаты распространяются по цепочке в обратном порядке

#### Сложности хореографии

| Проблема             | Описание                                                                                          |
|----------------------|---------------------------------------------------------------------------------------------------|
| **Координация API**  | Слишком сложно договориться об API между несколькими организациями                                |
| **Сетевые сбои**     | Если сервис изолирован по сети, он не может сообщить об ошибке предыдущему шагу                   |
| **Обработка ошибок** | Очень сложно обработать все кейсы, особенно когда сервисы падают или не передают статус об ошибке |

### Паттерн Saga: Оркестрация

**Принцип**: Централизованная координация через оркестратор.

```
Клиент → BFF → [Оркестратор] → Сервис 1
                             → Сервис 2
                             → Сервис 3
```

Оркестратор:
- Координирует последовательность вызовов
- Управляет успешными исходами и откатами
- Содержит всю логику транзакции в одном месте

#### Преимущества оркестрации

✅ **Наблюдаемость**: По логам удобно отслеживать весь процесс транзакции  
✅ **Контроль**: Централизованное управление всей логикой  
✅ **Простота кода**: Писать код оркестратора проще, чем распределенную логику  
✅ **Масштабируемость**: Можно масштабировать оркестратор, шардировать SQL базы  

#### Сложности оркестрации

⚠️ **Сложная логика**: Много условных переходов и вариантов обработки ошибок  
⚠️ **Точка отказа**: Оркестратор становится критичным компонентом системы  
⚠️ **Вероятность ошибок**: В сложной логике легко допустить ошибку

### Реализация оркестратора

#### Архитектура с WAL (Write-Ahead Log)

**Компоненты**:
1. **SQL база данных** — хранит журнал всех запросов оркестратора
2. **Write-Ahead Log** — перед выполнением операции записывается в лог
3. **Recovery механизм** — при падении оркестратор может восстановиться из лога

```
[Оркестратор] → [SQL WAL] → Сохранение состояния запроса
       ↓
   Выполнение запросов к сервисам
       ↓
   Обновление статуса в WAL
```

#### Ключевые возможности

**Надежность**:
- При падении оркестратор читает лог и продолжает выполнение с места остановки
- Все промежуточные состояния сохранены

**Мониторинг**:
- SQL база показывает, какие транзакции не завершились
- Можно создать инструменты для работы с проблемными запросами

**Масштабирование**:
- Оркестратор можно масштабировать горизонтально (увеличивать реплики)
- Можно шардировать SQL базу для распределения нагрузки

#### Пример схемы таблицы WAL

| Поле             | Тип          | Описание                                        |
| ---------------- |--------------| ----------------------------------------------- |
| `transaction_id` | UUID         | Уникальный идентификатор транзакции             |
| `status`         | ENUM         | Статус: PENDING, IN_PROGRESS, COMPLETED, FAILED |
| `current_step`   | INT          | Текущий шаг выполнения                          |
| `created_at`     | TIMESTAMP    | Время создания транзакции                       |
| `updated_at`     | TIMESTAMP    | Время последнего обновления                     |
| `payload`        | JSON         | Данные транзакции и шагов                       |

## Выводы и инсайты
- **Хореография vs Оркестрация**: Хореография подходит для простых цепочек в рамках одной организации, оркестрация — для сложных сценариев с множеством участников
- **WAL — основа надежности**: Write-Ahead Log позволяет оркестратору переживать сбои и гарантирует, что ни одна транзакция не потеряется
- **Компенсирующие операции обязательны**: Для каждой операции в Saga необходимо проектировать операцию отката
- **Наблюдаемость критична**: В распределенных транзакциях важно иметь полную видимость процесса — оркестрация в этом помогает лучше
- **Начните с оркестрации**: Для большинства задач оркестрация проще в реализации и поддержке, чем хореография
- **Масштабируемость через шардирование**: Оркестратор можно масштабировать как горизонтально (реплики), так и через шардирование базы данных
